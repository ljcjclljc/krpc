# W1-W2 开发改动文档

## 1. 文档用途
本文件用于记录本次针对 `W1/W2` 的代码落地内容，回答三件事：
1. 改了哪些文件，这些文件是做什么的。
2. 文件里的关键函数是做什么的。
3. 这次整体实现了什么，以及后续几周任务该如何写成可执行计划。

变更来源说明：本次改动基于你提出的“按修改后的 W1/W2 计划落实代码，并补充注释与文档”的需求执行。

## 2. 本次改动对应目标
- W1：补齐跨层接口抽象（`IRpcClient/IServiceDiscovery/ILoadBalancer`）和配置中心骨架（本地配置快照、更新、回滚）。
- W2：保留并整理协程状态机与调度能力（`READY/RUNNING/TERM`、`resume/yield`、生命周期回收）。
- 工程化：统一给核心文件增加 `Purpose` 文件头注释，新增并接入单元测试目标。

### 3.2 RPC 抽象层（W1）

| 文件 | 文件用途 | 关键函数/结构 |
| --- | --- | --- |
| `include/rpc/rpc/client.h` | 定义 RPC 请求模型和抽象接口，隔离网关与底层实现耦合 | `RpcRequest/RpcResponse/ServiceNode`、`IServiceDiscovery`、`ILoadBalancer`、`IRpcClient`、`init_client()`、`default_client()` |
| `src/rpc/rpc_stub.cpp` | 提供可运行的默认实现，验证接口抽象可用 | `InMemoryServiceDiscovery::list_nodes()`、`RoundRobinLoadBalancer::select_node()`、`DefaultRpcClient::invoke()`、`init_client()`、`default_client()` |

函数说明（核心）：
- `init_client(ClientInitOptions options)`：装配默认或外部注入的服务发现和负载均衡实现，创建全局默认客户端实例。
- `default_client()`：获取全局默认客户端；若未初始化则按默认依赖自动初始化。
- `DefaultRpcClient::invoke(const RpcRequest&)`：执行一次 RPC 调用流程（服务名校验 -> 获取节点 -> 负载均衡选点 -> 返回结果）。
- `RoundRobinLoadBalancer::select_node(...)`：轮询选择目标节点，保证同服务节点的基本均衡分发。
- `InMemoryServiceDiscovery::list_nodes(service)`：提供内存版服务发现，用于开发/测试阶段快速验证调用链。

### 3.3 Infra 配置治理骨架（W1）

| 文件 | 文件用途 | 关键函数/结构 |
| --- | --- | --- |
| `include/rpc/infra/infra.h` | 定义配置中心抽象与本地快照仓库接口 | `ConfigSnapshot`、`IConfigCenterClient`、`ConfigRepository`、`init_infra()`、`refresh_config_from_center()`、`config_repository()` |
| `src/infra/infra_stub.cpp` | 提供线程安全的最小实现（快照更新、历史记录、回滚） | `ConfigRepository::snapshot()`、`update_if_newer()`、`rollback_to()`、`history_versions()`、`init_infra()`、`refresh_config_from_center()` |

函数说明（核心）：
- `ConfigRepository::snapshot()`：读取当前配置快照（线程安全）。
- `ConfigRepository::update_if_newer(incoming)`：仅当版本更新时生效，避免旧配置覆盖新配置。
- `ConfigRepository::rollback_to(version)`：按历史版本回滚，支持快速恢复。
- `init_infra(InfraInitOptions options)`：初始化本地仓库与配置中心客户端，并执行一次拉取刷新。
- `refresh_config_from_center()`：向配置中心拉取最新快照并尝试更新本地版本。
- `config_repository()`：提供全局配置仓库只读访问入口。

### 3.4 Runtime 协程与调度（W2）

| 文件 | 文件用途 | 关键函数/结构 |
| --- | --- | --- |
| `include/rpc/runtime/coroutine.h` | 协程状态机与上下文切换接口定义 | `CoroutineState`、`Coroutine::resume()`、`yield()`、`yield_current()`、`run()` |
| `src/runtime/coroutine.cpp` | 协程切换实现（Windows Fiber / POSIX ucontext） | `resume/yield` 切换路径、异常到 `TERM` 的生命周期收口 |
| `include/rpc/runtime/scheduler.h` | 调度器接口定义，负责 READY 队列与回收 | `schedule()`、`run()`、`recycle_terminated()`、`alive_count()`、`completed_count()` |
| `src/runtime/scheduler.cpp` | 调度主循环与协程 ID 回收实现 | `allocate_id()`、`recycle()`、`run()` |

函数说明（核心）：
- `Coroutine::resume()`：把 `READY` 协程切入执行，状态进入 `RUNNING`。
- `Coroutine::yield()`：当前协程主动让出执行权，状态回到 `READY`。
- `Coroutine::yield_current()`：当前协程静态让出入口，便于业务代码调用。
- `Coroutine::run()`：执行用户回调，并在正常/异常路径统一进入 `TERM`。
- `CoroutineScheduler::schedule()`：创建协程并加入 READY 队列。
- `CoroutineScheduler::run()`：循环调度 READY 协程，按状态决定重入队或回收。
- `CoroutineScheduler::recycle_terminated()`：批量回收已结束协程，释放资源并记录完成计数。

## 5. 后续几周任务怎么写（建议）

### 5.1 周任务写作模板（建议统一格式）
每周都按下面 6 段写，避免“只有任务、没有验收”：
1. **周目标**：一句话描述本周要达成的能力。
2. **开发任务**：3-5 条，动作可落地（实现/接入/补齐/清理）。
3. **改动范围**：明确目录和核心文件。
4. **验收标准**：至少 2 条，必须可量化或可执行（测试名、指标阈值、故障场景）。
5. **风险与回滚**：本周高风险点 + 失败时回退策略。
6. **交付物**：代码、测试、文档、脚本分别产出什么。

### 5.2 W3-W8 任务示例（可直接放进周计划）

#### W3（M:N 调度器）
- 周目标：从单线程调度扩展到多线程 Worker 调度模型。
- 开发任务：
1. 实现 Worker 线程池与任务分发。
2. READY 队列改为线程安全结构。
3. 增加 `stop/join` 生命周期控制。
- 验收标准：
1. 多线程压力下无死锁。
2. 调度吞吐较 W2 基线有提升并可复现。

#### W4（IO 事件循环）
- 周目标：打通 Epoll 事件到协程恢复链路。
- 开发任务：
1. 实现 `FdContext` 读写事件注册/注销。
2. 建立 IO 回调与协程恢复绑定。
3. 加入异常事件清理（`EPOLLERR/EPOLLHUP`）。
- 验收标准：
1. 读写事件触发后协程恢复正确。
2. 异常断链场景无 fd 泄漏。

#### W5（定时器与 deadline）
- 周目标：形成 IO + Timer 统一调度和超时语义。
- 开发任务：
1. 完成最小堆定时器管理。
2. 为请求上下文增加 deadline 传递。
3. 补齐超时取消路径与错误码映射。
- 验收标准：
1. 定时器精度满足预期。
2. 超时触发后请求能及时退出且可观测。

#### W6（Hook 首批）
- 周目标：将常见阻塞调用协程化。
- 开发任务：
1. Hook `sleep/read/write/recv/send`。
2. 增加 TLS Hook 开关。
3. 对齐 `errno` 语义和边界行为。
- 验收标准：
1. Hook 开启时阻塞调用可异步化。
2. Hook 关闭时行为与原生一致。

#### W7（超时分层 + 重试预算）
- 周目标：上线基础稳定性治理。
- 开发任务：
1. 打通 gateway -> rpc -> net 超时预算传递。
2. 实现重试预算窗口与限制策略。
3. 增加预算命中指标埋点。
- 验收标准：
1. 不出现无限重试。
2. 故障注入下无明显级联放大。

#### W8（背压与快速失败）
- 周目标：引入过载保护，避免高峰雪崩。
- 开发任务：
1. 接入入口有界队列上限。
2. 设置连接池等待上限和超时。
3. 超限直接返回 `429/503`。
- 验收标准：
1. 峰值流量下系统可退化但不崩溃。
2. 过载指标与错误码可观测。

## 6. 建议的提交与文档联动方式
1. 每周创建一个 `docs/wX.md`，按“5.1 模板”记录实施与验收结果。
2. 每次代码提交需同步更新对应 `docs/wX.md` 的“改动范围”和“验收标准”。
3. 每个新文件必须保留 `Purpose` 头注释，函数注释只写关键流程和边界条件。
