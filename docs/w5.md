## 4. 关键设计与实现

### 4.1 最小堆定时器（TimerManager）
核心结构：
1. 最小堆：按 `expires_at`（同时间按 `id`）维护最近到期任务。
2. 哈希表：`timer_id -> TimerNode`，用于 O(1) 查找和取消。
3. 取消标记：`cancel_timer()` 只标记 `cancelled`，不直接在堆中删除节点。

关键接口：
1. `add_timer(delay, callback)`：计算到期时间，分配唯一 ID，插入堆和哈希表。
2. `cancel_timer(id)`：通过哈希表定位并标记取消，延迟到堆顶清理时移除。
3. `collect_expired_callbacks()`：循环弹出到期/已取消节点，返回待执行回调。
4. `next_timeout_ms()`：返回 `-1/0/>0`，供 `epoll_wait` 直接使用。

### 4.2 IOManager 与 Timer 融合
核心改动：
1. `IOManager` 新增 `add_timer/cancel_timer`，内部持有 `TimerManager`。
2. `event_loop` 每轮先取 `timer_manager_.next_timeout_ms()` 作为 `epoll_wait` 超时。
3. `epoll_wait` 返回后统一调用 `collect_expired_callbacks()`，并通过调度器执行。
4. 新增 `eventfd` 唤醒机制：
- 新增更早到期定时器或取消定时器时，主动唤醒 `epoll_wait` 线程。
- `stop()` 时也通过唤醒 FD 让事件线程快速退出。
5. 线程停止时清空全部定时器，避免残留回调。

### 4.3 请求 deadline 上下文基础结构
新增 `RequestContext`：
1. 基础字段：`request_id`、`deadline`、`cancelled`、`cancel_reason`、业务键值数据。
2. 超时绑定：`bind_deadline_timer(IOManager&)` 注册到期回调（自动 `cancel`）。
3. 轮询检查：`check_deadline_and_cancel()` 支持业务逻辑周期性检查。
4. TLS 传递：
- `set_current_request_context()` / `current_request_context()`
- `ScopedRequestContext` 作用域绑定与自动恢复。

### 10.2 高频问答
1. 问：为什么取消定时器不立即从堆中随机删除。  
答：随机删除会增加堆维护复杂度，延迟删除可保持实现简单且性能稳定。
2. 问：为什么要用 `eventfd` 主动唤醒。  
答：否则 `epoll_wait` 可能睡到旧超时点，新增更早任务会被延后处理。
3. 问：`RequestContext` 为什么要放 TLS。  
答：减少跨层函数签名污染，并保证同线程/协程可就近读取请求上下文。
