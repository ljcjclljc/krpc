## 1. 项目概述与定位

### 1.1 项目背景
本项目以 C++17 为核心，目标不是“单点功能 Demo”，而是构建可演进、可治理、可观测、可发布的高性能网关与 RPC 运行平台。项目覆盖从运行时、网络层、网关层到一致性基础设施的完整技术栈。

### 1.2 项目定位
1. 作为网关入口：承接 HTTP 请求、完成鉴权、路由、治理与回包。  
2. 作为 RPC 平台：支持服务发现、负载均衡、重试预算、熔断、灰度与降级。  
3. 作为稳定性底座：具备协程调度、IO 事件循环、定时器、Hook、配置热更新、证书轮转、Raft 一致性元数据能力。  
4. 作为工程化样板：从代码注释规范、单元测试、压测到发布冻结形成闭环。

### 1.3 当前版本与范围
1. 工程版本基线：`0.1.0`。  
2. 阶段覆盖：`W1` 到 `W18`。  
3. 产物形态：核心库 `rpc_core` + 可执行程序 `rpc_gateway_app` + 多组单测目标。

## 2. 整体架构与分层设计

### 2.1 分层架构
```text
[Client]
   |
HTTP
   v
[net/reactor_http_server]
   |
GatewayHttpRequest
   v
[gateway/http_gateway]
   |
RpcRequest
   v
[rpc/client + krpc_channel]
   |
NetCallRequest
   v
[net/net_stub invoke_tcp/TLS]
   |
Runtime/IO/Timer/Hook
   v
[runtime/*]

Infra 横切：
[infra/config + memory_pool + structured_log + raft]
```

### 2.2 模块职责（与代码一致）
1. `runtime`：协程、M:N 调度、IOManager、Timer、Hook、请求上下文。  
2. `net`：Reactor HTTP 服务、下游 TCP/TLS 调用、限流与背压。  
3. `gateway`：Router/Servlet、鉴权、错误映射、Trace 与路由指标。  
4. `rpc`：RPC 抽象、KrpcChannel 编解码、自适应 LB、熔断降级、灰度。  
5. `infra`：配置快照与回滚、内存池、结构化日志、Raft 一致性。

### 2.3 启动顺序
主程序按以下顺序初始化：
1. `infra::init_infra()`  
2. `runtime::init_runtime()`  
3. `net::init_network()`  
4. `gateway::init_gateway()`  
5. `client::init_client()`

该顺序保证：配置与基础设施先可用，运行时与网络能力先就绪，再挂载网关与 RPC 客户端。

## 3. 核心技术深度讲解

## 3.1 协程与调度（W2/W3）

### 3.1.1 协程状态机
协程状态定义为：
1. `READY`：可调度。  
2. `RUNNING`：执行中。  
3. `WAITING`：等待外部事件（IO/定时器）。  
4. `TERM`：结束。

### 3.1.2 调度模型
1. `CoroutineScheduler` 提供 M:N 调度。  
2. worker 线程拥有本地队列 + 本地快队列；外部提交先入全局队列。  
3. 提供工作窃取（`steal_count`）提升负载均衡。  
4. 提供 `pending/active/alive/completed` 等运行态指标。

### 3.1.3 并发语义增强
1. `resume(id)` 仅将 `WAITING -> READY`，防止非法状态恢复。  
2. 对“恢复早于挂起”场景使用 `pending_resumes_` 处理竞态。  
3. 锁等待纳入剖面统计，便于性能调优。

## 3.2 IO 事件循环与定时器（W4/W5）

### 3.2.1 IOManager
1. 封装 `epoll` 事件管理。  
2. `FdContext` 管理读/写事件绑定与触发。  
3. 回调触发后通过调度器执行，统一执行模型。

### 3.2.2 TimerManager
1. 最小堆维护到期任务。  
2. `next_timeout_ms()` 直接用于 `epoll_wait` 超时。  
3. `eventfd` 唤醒机制保证新更早定时器可立即生效。  
4. 支持取消与惰性清理。

### 3.2.3 RequestContext 与 deadline
1. 支持 `request_id/deadline/cancelled/cancel_reason`。  
2. TLS 作用域传递（`ScopedRequestContext`）。  
3. 可绑定 deadline 定时取消。

## 3.3 Hook 机制（W6）

### 3.3.1 覆盖能力
首批 Hook：`sleep/read/write/recv/send`。

### 3.3.2 生效条件
仅当以下条件同时满足才走 Hook 路径：
1. TLS Hook 开关开启。  
2. 当前在协程上下文。  
3. 已绑定可用 `IOManager`。

### 3.3.3 行为语义
1. `EINTR` 重试。  
2. `EAGAIN/EWOULDBLOCK` 注册事件并让出协程后重试。  
3. 非 Hook 场景保持原生语义，兼容 `errno`。

## 3.4 网关入口与 Reactor（W8/W9/W10）

### 3.4.1 HTTP 解析与主循环
1. 增量解析请求行、Header、Body。  
2. `accept/read/write` 全部走 epoll Reactor。  
3. Keep-Alive 与短连接场景均可处理。

### 3.4.2 背压与快速失败
1. 入口有界队列容量限制。  
2. 连接数上限与连接内在途请求上限。  
3. 队列满返回 `429`，连接超限返回 `503`。  
4. 写缓冲上限防止单连接挤占内存。

### 3.4.3 路由、鉴权、错误映射
1. `Router + Servlet` 支持精确与前缀路由。  
2. `AuthInterceptor` 支持 API Key/JWT（OR 语义）。  
3. `map_rpc_code_to_http_status()` 统一 RPC->HTTP 错误语义。

### 3.4.4 Trace 与指标
1. 自动生成或透传 `x-trace-id`。  
2. 路由级指标：请求数、错误率、`p50/p90/p99`。

## 3.5 RPC 调用治理（W7/W11/W13/W16）

### 3.5.1 调用抽象
1. `IServiceDiscovery`  
2. `ILoadBalancer`  
3. `IRpcClient`

上层网关只依赖抽象接口，避免跨层耦合。

### 3.5.2 KrpcChannel
1. 提供同步 `invoke` 与异步 `invoke_async`。  
2. 支持 Raw/Protobuf 双模式。  
3. Protobuf 使用 `ByteSizeLong + SerializeToArray`，并支持线程局部对象复用。

### 3.5.3 超时预算与重试预算
1. 生效超时：`effective_timeout = min(上游预算, 下游配置)`。  
2. 重试预算采用滑动窗口 token，预算耗尽返回 `429 retry_budget_exhausted`。  
3. 观测字段包括 attempts/effective_timeout/预算快照。

### 3.5.4 自适应 LB、熔断、降级、灰度
1. 自适应 LB 基于 CPU/内存/QPS/延迟与失败惩罚打分。  
2. 熔断状态机：`closed/open/half-open`。  
3. 降级策略：缓存兜底、静态兜底。  
4. 灰度策略：标签直达 + 用户哈希比例。

## 3.6 Raft 一致性元数据（W12）

### 3.6.1 核心能力
1. Leader 选举。  
2. 日志复制与提交。  
3. 快照裁剪与安装。  
4. Joint Consensus 在线扩缩容。

### 3.6.2 元数据承载
1. 配置同步（config key/value）。  
2. 服务注册发现（service -> instances）。

### 3.6.3 工程化增强
1. WAL 与快照持久化恢复。  
2. Raft RPC 传输封装（投票/追加/快照）。  
3. 故障注入：丢包、延迟抖动、节点上下线演练。

## 3.7 内存治理与结构化日志（W14/W15）

### 3.7.1 分代内存池
1. Short/Long 双代池。  
2. 热扩容 + 冷缩容。  
3. 越界 guard 与泄漏检测。  
4. 指标：命中率、高水位、锁等待。

### 3.7.2 结构化日志
1. JSON 行输出。  
2. 级别采样。  
3. 慢请求强制采样。  
4. 采样统计（emitted/dropped/forced）。

## 4. 全链路运行逻辑（从启动到请求完成）

## 4.1 服务启动逻辑
1. 读取/初始化配置快照。  
2. 启动 runtime（scheduler + IOManager）。  
3. 启动 Reactor HTTP Server（监听 socket + epoll + worker）。  
4. 挂载 Gateway 内建路由与 RPC 客户端。

## 4.2 HTTP 请求处理逻辑
1. `accept` 新连接，超上限直接 503。  
2. 非阻塞 `read`，增量 parser 解析。  
3. 解析通过后入 ingress 队列，队列满返回 429。  
4. worker 执行 `HttpGateway::handle()`。

## 4.3 网关处理逻辑
1. 归一化 method/uri。  
2. 路由匹配（先精确后前缀）。  
3. 路由要求鉴权时执行 API Key/JWT 校验。  
4. 构造 `RpcRequest`（包含 trace、灰度、治理参数）。  
5. 调用 `IRpcClient::invoke()`，拿到 `RpcResponse`。  
6. 统一映射 HTTP 状态码并封装回包。

## 4.4 RPC 调用逻辑
1. 服务发现拿节点。  
2. 负载均衡选节点。  
3. 根据治理参数计算超时、熔断、重试。  
4. 通过 `KrpcChannel` 编解码并下沉 `net::invoke_tcp`。  
5. 将结果反馈给 LB 运行态（失败率/EWMA/熔断状态）。

## 4.5 Net 调用逻辑（含 TLS）
1. 解析 endpoint（host:port）。  
2. 计算有效超时并设置 socket 超时。  
3. 若 TLS 开启，加载/复用 SSL context，进行握手。  
4. mTLS 场景加载客户端证书。  
5. 返回 `NetCallResponse`，并上抛取消/超时语义。

## 4.6 配置热更新与回滚逻辑
1. `publish_config_snapshot` 或 `publish_config_patch` 发布新版本。  
2. 运行中模块按 `config_version` 感知并刷新（如限流、TLS）。  
3. 异常时执行 `rollback_config_to(version)` 回退。  
4. 通过版本与指标确认恢复完成。

## 4.7 运行时协程化等待逻辑
1. Hook 场景将阻塞调用转换为事件等待。  
2. 协程 `yield_waiting()` 进入 WAITING。  
3. IO 或 Timer 触发后 `resume` 回 READY。  
4. 调度器重入执行并继续流程。

## 5. 项目特点与核心优点（全量）

### 5.1 架构层优点
1. 分层清晰、接口先行，上层不直接依赖下层实现细节。  
2. 横切能力（配置、日志、内存池、Raft）统一收敛到 infra。  
3. `rpc_core` 聚合构建减少目标间隐式依赖扩散。

### 5.2 运行时优点
1. M:N 调度提升线程利用率。  
2. IO+Timer 单等待点降低复杂度。  
3. Hook 保持非 Hook 兼容语义，迁移成本低。

### 5.3 网关治理优点
1. 背压与快速失败策略明确（429/503 分工清晰）。  
2. 路由、鉴权、错误映射形成稳定处理链。  
3. 路由级指标 + trace 便于快速排障。

### 5.4 RPC 治理优点
1. 超时预算与重试预算双治理，防止故障放大。  
2. 自适应 LB + 熔断 + 降级 + 灰度形成组合稳定性策略。  
3. 调用反馈闭环使负载策略具备动态收敛能力。

### 5.5 稳定性与可靠性优点
1. 配置热更新不重启。  
2. 证书轮转失败可回退旧 context，避免全量不可用。  
3. Raft 提供一致性元数据能力，减少外部强依赖。

### 5.6 工程化优点
1. 单测覆盖完整，分层与集成场景兼备。  
2. 性能剖面指标可用于持续优化。  
3. 发布基线与 tag 冻结流程明确，可复现构建。

## 6. 源文件写作规范对照与统一模板

## 6.1 当前仓库已采用的规范（从源文件归纳）
1. 大多数 `.h/.cpp` 文件头都有“文件用途 / File purpose”注释。  
2. 关键函数前有简短中文注释，聚焦流程与边界，不写冗余描述。  
3. 头文件定义抽象与模型，源文件落地实现，职责分离清晰。  
4. `namespace` 分层与目录分层一致，便于定位。

## 6.2 与参考“终极版”写法对齐点
1. 均强调“模块定位 -> 流程 -> 优缺点 -> 演进方向”。  
2. 均按链路视角讲解（非纯 API 罗列）。  
3. 均重视面向复盘与面试复述的表达。

## 6.3 发现的规范差异与建议
1. `include/rpc/runtime/scheduler.h` 与 `src/runtime/scheduler.cpp` 头部未写“文件用途”说明，可补齐。  
2. 少量文件中英文风格混用（`File purpose` 与 `文件用途`），建议统一团队规范。  
3. 部分复杂函数已有注释但可进一步增加“输入约束/失败语义”短注释。

## 6.4 建议统一模板（可直接复用）

## 附录：面试与汇报高频回答框架

1. 为什么要“预算治理”而不是只做 `max_retries`？  
答：`max_retries` 只能约束单请求，预算机制约束系统级重试放大。

2. 为什么要把 429 和 503 分开？  
答：429 表示治理策略拒绝（可退避重试），503 表示资源或后端不可用（策略不同）。

3. 为什么网关只依赖 `IRpcClient`？  
答：隔离实现细节，替换传输/LB/发现时上层不改。

4. 为什么需要配置回滚与证书轮转兜底？  
答：线上变更不可避免，必须保证“分钟级恢复”和“失败不扩散”。

5. Raft 对本项目最大的价值是什么？  
答：把配置与服务发现的一致性能力内建化，减少外部依赖与运维复杂度。
