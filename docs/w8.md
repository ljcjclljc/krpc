## 4. 关键设计

### 4.1 HTTP parser（增量解析 + 限制）
1. 解析范围：`request line + headers + body(Content-Length)`。
2. 解析策略：在连接读缓冲上按前缀解析，返回 `NeedMoreData/MessageComplete/BadRequest/HeaderTooLarge/BodyTooLarge`。
3. 限制策略：
- Header 超限返回 `431 Request Header Fields Too Large`。
- Body（Content-Length）超限返回 `413 Payload Too Large`。
4. Keep-Alive 语义：
- HTTP/1.1 默认 keep-alive，`Connection: close` 关闭。
- HTTP/1.0 默认 close，`Connection: keep-alive` 开启。

### 4.2 Reactor 主循环（accept/read/write）
1. 事件源：监听 fd + notify eventfd + 所有连接 fd，统一由 epoll 驱动。
2. `accept`：连接数超过 `max_connections` 时直接返回 `503` 并关闭连接。
3. `read`：非阻塞读取 -> 解析请求；支持同连接多请求（短连/长连）。
4. `write`：非阻塞发送写缓冲，写空后根据 keep-alive 与 in-flight 状态决定保活或关闭。

### 4.3 有界队列与快速失败
1. 入口队列：`ingress_queue_capacity` 限制请求入队数量。
2. 队列满：直接返回 `429 Too Many Requests`（不进入 worker）。
3. 连接内在途上限：`max_connection_requests_inflight`，超限返回 `503`。
4. 写缓冲上限：`max_connection_write_buffer_bytes`，防止单连接输出内存无界增长。

### 4.4 峰值保护与可观测
新增统计项：
1. `ingress_queue_size / ingress_queue_high_watermark`
2. `read_buffer_bytes / write_buffer_bytes`
3. `read_buffer_high_watermark / write_buffer_high_watermark`
4. `rejected_429 / rejected_503`
5. `parser_400 / parser_413 / parser_431`

这些指标可用于验证“峰值流量下无队列失控和内存暴涨”。

## 9. 复习提纲（建议背诵）

### 9.1 必背概念
1. `max_connections`、`ingress_queue_capacity`、`max_connection_requests_inflight` 三层限额分工。
2. 429 与 503 在背压场景下的语义边界。
3. Header/Body 限制在解析早期拦截的性能收益。

### 9.2 高频问答
1. 问：为什么队列满直接返回 429，而不是继续排队。  
答：继续排队会放大延迟和内存占用，快速失败可保护系统核心吞吐。
2. 问：为什么同一连接还需要在途请求上限。  
答：防止单连接长期占用 worker 与写缓冲，影响全局公平性。
3. 问：为什么需要高水位指标而不只看瞬时值。  
答：高水位更能反映峰值压力和容量边界，适合做稳定性评估。

1. 能否解释一次请求从 `accept -> read/parse -> enqueue -> worker -> write` 的完整路径。
2. 能否说明在哪些阶段可能返回 413、431、429、503。
3. 能否基于 `ingress_queue_high_watermark` 推导容量是否接近瓶颈。
