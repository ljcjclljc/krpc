# 《RPC 网关项目极限拷问题库（从深到浅）》

说明：每题都包含主问，并在括号里给出`八股方向`和`场景题`，按大厂“往死里扣”风格设计。

## 一、深级（架构与方法论）
1. 你这套系统最核心的不变量是什么，哪些模块绝不能绕开抽象直接调用？（八股：分层架构/依赖倒置；场景：线上紧急修 bug，开发想直接跨层取内部状态）
这套系统最核心的不变量是：依赖方向不能反、状态语义不能乱、配置版本不能倒退。
具体我会守三条红线：

分层依赖固定：Gateway -> RPC抽象 -> Net/Runtime/Infra，上层不能绕过抽象直连下层实现。

运行时状态机固定：协程只能按 READY -> RUNNING -> WAITING/READY -> TERM 合法流转，不能跨状态乱 resume/yield。

配置治理语义固定：update_if_newer 只接收新版本，回滚必须走标准接口，不能直接改内部仓库状态。

绝不能绕开抽象直接调用的模块边界：

gateway 只能通过 IRpcClient 调上游，不能直接读 net 连接内部状态、也不能直接摸 infra 配置内部结构。

业务层不能直接操作 Coroutine/CoroutineScheduler/IOManager/FdContext 这些 runtime 内部对象，只能走 runtime.h 外观接口。

上层不能直接依赖 infra 的内部容器和锁，只能走 IConfigCenterClient/ConfigRepository 对外接口。

即使线上紧急修 bug，也不建议跨层取内部状态；正确做法是让“状态所属层”补一个只读快照接口，通过抽象暴露，修完再回收临时观测点。


2. 为什么先做 W1 抽象再做功能，不先写“能跑”的业务代码？（八股：抽象与实现分离；场景：需求频繁变更导致重构成本爆炸）
面试回答：先做 W1 抽象，是为了先把跨层契约钉死，避免需求一变就全链路重构。接口先行后，功能迭代只是在实现层替换，不会把网关、RPC、配置治理一起拖着改。短期看“慢一点”，长期看回归成本和线上风险都更低。
术语解释：抽象与实现分离=先定义“怎么调用”，再决定“怎么实现”；重构爆炸=一个需求改动引发多层级联修改。

术语解释：抽象与实现分离=先定义“怎么调用”，再决定“怎么实现”；重构爆炸=一个需求改动引发多层级联修改。
3. 你如何证明这个项目是“可演进系统”而不是“功能堆叠系统”？（八股：可扩展性/技术债；场景：半年后加协议、换注册中心）
面试回答：看三点：第一，替换能力，网关只依赖 IRpcClient，可换发现、LB、传输实现。第二，演进证据，W1 到 W18 每周都有目标、验收、回滚策略，不是临时打补丁。第三，回归护栏，核心能力都有测试和快照指标，改动可验证。
术语解释：可演进=需求变化时主要改“实现”而不是“调用方”；技术债=今天快写但未来每次改都更贵的问题。
4. 从 `HTTP -> Gateway -> RPC -> Net -> Runtime`，哪一层是系统瓶颈第一嫌疑，为什么？（八股：性能分析方法；场景：QPS 掉 40% 但 CPU 不高）
面试回答：在“QPS 掉 40% 但 CPU 不高”场景，我先怀疑 Net/Runtime 交界，不是业务算力不够，而是等待/阻塞/队列拥塞。先看 ingress 队列高水位、pending_tasks、epoll 变更和超时分布，再判断是网络等待还是调度拥塞。
术语解释：瓶颈嫌疑=最可能先出问题的层；高水位=一段时间内的峰值，不是瞬时值。
5. 你的错误语义设计里，`429/503/504/502` 的边界为什么这样划？（八股：错误码分层；场景：客户端误把 503 当可无限重试）
面试回答：429 是“策略拒绝”，如限流或重试预算不足；503 是“服务当前不可用”，如过载或熔断打开；504 是“超时”，预算耗尽或下游超时；502 是“上游返回不可归类坏响应”，网关兜底。这样客户端才知道该限速、重试还是告警。
术语解释：错误语义分层=同样失败但原因不同，状态码要反映处理策略；兜底=无法精确分类时的保守返回。
6. 你的“稳定性治理闭环”具体由哪些机制组成，如何避免互相打架？（八股：超时/重试/熔断/限流；场景：超时和重试叠加导致雪崩）
面试回答：闭环是“超时预算 + 重试预算 + 熔断 + 限流背压 + 降级 + 配置回滚 + 观测”。避免打架靠统一决策顺序：先判预算/限流，再判熔断，再执行调用与重试，最后走降级；每一步有明确停止条件，不允许无限重试。
术语解释：闭环=发现问题、限制扩散、恢复和验证形成完整链条；背压=系统忙时主动拒绝或减速，避免雪崩。
7. 你如何定义“成功请求”和“健康系统”，指标口径在哪层统一？（八股：SLI/SLO；场景：网关 200 但下游业务失败）
外部口径在 Gateway 统一：HTTP 2xx 仅代表网关处理成功，不等于业务成功。内部口径要叠加 RPC 结果（如 code=0）和超时/降级标记；健康系统看成功率、延迟分位、错误结构和资源水位是否稳定。
术语解释：SLI=可量化的服务指标；SLO=对指标的目标承诺；口径统一=所有团队按同一规则算成功/失败。

8. 你如何证明默认配置是“安全默认值”而不是“拍脑袋参数”？（八股：配置治理；场景：新环境一上线就误熔断）
不是拍参数，而是“保守默认 + 测试验证 + 可回滚”。比如默认关闭高风险开关、超时和重试有硬上限、限流有边界；再用压测和故障注入验证；上线后靠版本化配置和回滚在分钟级止损。
术语解释：安全默认值=即使用户不懂配置，也不容易把系统直接打崩；版本化配置=每次配置变更都有版本号和回退点。
9. 这套系统如果必须删一个模块保命，你删谁，代价是什么？（八股：系统裁剪/降级设计；场景：内存资源严重不足）
面试回答：我先删（或关闭）Hook 协程化路径，不删主链路抽象。因为它是性能增强，不是协议正确性的硬依赖；关掉后系统还能跑，但吞吐下降、线程占用上升、尾延迟会变差。
术语解释：保命裁剪=先保可用性，再保性能；尾延迟=最慢那部分请求的延迟（如 p99）。
10. 你设计中最容易被误用的接口是哪一个，怎么防误用？（八股：API 设计；场景：协程外调用 `yield_current`）
面试回答：最容易误用的是 yield_current()/yield_current_waiting()，在非协程上下文调用会破坏运行语义。防法是三层：运行时强校验直接报错、业务层只暴露 runtime 外观接口、测试覆盖误用场景。
术语解释：API 误用=接口能调用但调用时机不对；外观接口（Facade）=给业务一个更安全、更窄的调用面。
11. 如果要支持多租户隔离，你先改哪三处？（八股：隔离模型；场景：一个租户流量打爆全局）
面试回答：第一，Gateway 做租户级限流和队列配额，防止单租户打爆全局。第二，RPC 治理做租户级超时/重试/熔断预算。第三，配置和观测按租户分命名空间，做到可配置、可审计、可追责。
术语解释：多租户隔离=不同租户互不影响；命名空间=把配置、指标、策略按租户分桶管理。
12. 你认为当前版本离生产级还差哪 3 个阻断项？（八股：上线门禁；场景：老板要求下周灰度）
面试回答：第一，安全阻断：JWT 当前仅校验 claim，不验签，不满足生产认证要求。第二，发布阻断：发布脚本、签名、SBOM、依赖审计门禁还不完整。第三，可靠性阻断：长稳压测与故障演练的 CI 门禁还不够强。
术语解释：阻断项=不解决就不该上线的硬条件；SBOM=软件物料清单，用于依赖追踪和合规审计。

## 二、全链路流程死扣（端到端）
1. 从 `accept` 到响应写回，逐函数说出关键状态转移点。`Connection` 结构里哪些字段最关键？（八股：Reactor 模型；场景：长连接随机断）
面试回答：accept -> handle_accept 建连并注册 EPOLLIN，handle_read 读包并用 parser 做“未完成/完成/错误”分流；完成后入 ingress_queue 且 inflight_requests++。worker_loop -> process_request 产出响应，enqueue_completed -> handle_completed_tasks 回填写缓冲并 inflight_requests--；handle_write 按 write_offset 发送，发送完且 close_after_write && inflight==0 才 close_connection。Connection 最关键字段是 fd/stream/request_parser/write_buffer/write_offset/inflight_requests/close_after_write/write_interest_enabled。
术语解释：Reactor=一个线程专管 IO 事件分发；状态转移点=对象从“可读/可写/待处理/待关闭”变化的关键时刻。
2. 请求在 ingress 队列满时为何返回 429 而不是排队？（八股：背压策略；场景：峰值 10 倍突发）
面试回答：队列满返回 429 是主动背压，不是“多排一会儿”。10 倍突发时继续排队会把延迟和内存一起打爆，最后变成全局雪崩。
术语解释：背压=系统忙时快速拒绝，保护整体可用性；雪崩=局部过载扩散成全链路不可用。
3. `keep-alive` 与 `close_after_write` 的协同细节是什么？（八股：HTTP 连接复用；场景：响应已发但连接没关导致 fd 泄漏）
keep-alive 决定“协议上能不能复用连接”，close_after_write 决定“实现上何时真正关闭 fd”。实现是先写完缓冲，再在 inflight==0 条件下关连接，避免“响应写了一半就关”或“永远不关 fd 泄漏”。
术语解释：keep-alive=HTTP 长连接复用；fd 泄漏=连接对象不释放，句柄越积越多。
4. 路由命中顺序是精确优先还是前缀优先，为什么？（八股：路由算法；场景：`/rpc/*` 抢占 `/rpc/invoke`）
路由是“精确优先，再前缀”，并且前缀按长度降序匹配。这样 /rpc/invoke 不会被 /rpc/* 抢占。
术语解释：精确匹配=路径完全相同才命中；前缀匹配=满足某个路径开头即可。
5. TraceId 生成与透传在哪层落地，如何保证上下游日志能串？（八股：可观测性；场景：一次请求跨层排障）
TraceId/SpanId 在 Gateway 入口生成或补齐，然后透传到 RPC metadata（x-trace-id/x-span-id）。网关请求日志和 RPC 调用日志都带同一 trace 字段，跨层就能串起来。
术语解释：TraceId=一次请求全链路身份证；透传=下游继续携带上游传来的上下文。
6. 什么时候会走 `map_rpc_code_to_http_status` 的兜底 502？（八股：网关错误映射；场景：上游返回 599）
map_rpc_code_to_http_status 里，0->200，4xx透传，503/504透传，其他 5xx（比如 599）统一兜底 502。也就是“上游失败但语义不标准”时，网关给出一致的坏网关语义。
术语解释：兜底=无法细分时的统一保守策略；坏网关=网关与上游交互异常。
7. 网关何时做鉴权，为什么要前置在业务处理前？（八股：责任链；场景：无权限请求耗尽 worker）
鉴权在“路由命中后、业务处理前”执行，且仅对 auth_required 路由生效。前置能挡掉无权限流量，避免它占满 worker 和下游 RPC。
术语解释：责任链=按固定顺序执行的一组处理环节；前置鉴权=先验身份再执行业务。
8. `runtime_scheduler_snapshot` 里哪些字段可以推断调度拥塞？（八股：运行时观测；场景：pending 长期不降）
看拥塞先看 pending_tasks 是否长期高位不降，再看 alive_coroutines 是否持续堆积。同时结合 idle_switches（空闲切换）下降、锁等待统计上升，可判断是调度器忙还是线程空转。
术语解释：拥塞=任务进入速度大于处理速度；高位不降=系统一直“欠账”没还完。
9. 你如何判定 timeout 是上游预算问题还是下游慢？（八股：超时分层；场景：同请求不同环境 timeout 不一致）
面试回答：先看“有效超时”是被上游预算压扁，还是被下游慢调用耗尽。若剩余 deadline 很小或已过期，根因是上游预算；若上游预算充足但下游多次打满超时，根因是下游慢。
术语解释：超时分层=不同层各自有时间预算；deadline=整个请求生命周期的总时限。
10. `await_io` 里 IO 回调与 timeout 回调竞态怎么收敛？（八股：CAS/竞态控制；场景：同时触发 ready 和 timeout）
await_io 用一个原子状态 0(pending)/1(ready)/2(timeout)，IO 回调和定时器回调都用 CAS 抢占，只有第一个成功者能 resume 协程。唤醒后取消另一路，超时分支还会 del_event，避免重复触发。
术语解释：CAS=“仅当值未变化才写入”的原子操作；竞态收敛=多个同时事件最终只认一个结果。
11. Hook 开关为什么是 TLS，而不是全局原子变量？（八股：线程隔离；场景：非协程线程行为被污染）
Hook 开关做成 TLS（线程局部）是为了隔离线程行为，避免一个线程开 Hook 把其他线程也污染。全局原子会导致非协程线程也走 Hook 路径，语义和风险都不可控。
术语解释：TLS 这里是 Thread Local Storage（线程局部存储），不是传输层安全协议；线程污染=本不该受影响的线程被全局开关改变行为。


12. `epoll_mod_calls` 和 `epoll_mod_skipped` 指标要一起看什么问题？（八股：系统调用优化；场景：syscall 激增）
这两个指标要一起看“无效 epoll_ctl MOD 是否被优化掉”。calls 高且 skipped 低，说明频繁改兴趣集可能有 syscall 浪费；skipped 高说明优化在生效。
术语解释：兴趣集=epoll 当前关心的读写事件集合；syscall 浪费=频繁进内核但收益小。
13. 为什么要把 `add_event` 回调交给调度器执行，而不是事件线程直接执行？（八股：线程模型；场景：事件线程被业务回调阻塞）
面试回答：事件线程应该只做“快操作”（收包、分发、写回），业务回调放到调度器执行。否则一个慢回调就能阻塞 epoll_wait，把全体连接拖慢。
术语解释：线程模型=线程职责分工方式；头阻塞=前面的慢任务卡住后面的所有任务。
14. 连接写缓冲上限的核心保护对象是谁：单连接还是全局？（八股：资源隔离；场景：慢客户端拖垮服务）

术语解释：资源隔离=坏连接/坏租户不能拖垮全局；慢客户端=读得慢导致服务端写缓冲堆积。
15. 全链路里哪个步骤最容易造成“看似成功但语义失败”？（八股：一致性与语义正确性；场景：HTTP 200 + 业务失败码）
最容易“HTTP 200 但语义失败”的点在 Gateway-RPC 语义边界：如果上游把业务失败塞进 payload 但 rpc_code 仍是 0，网关会映射成 200。治理办法是强制统一响应契约（业务码与传输码分离）并做“200+失败业务码”告警。
术语解释：语义失败=协议成功但业务没成功；契约=上下游共同遵守的数据/状态约定。

## 三、Runtime（协程/调度/IO/Hook）逐功能拷问
1. `CoroutineState` 为什么要有 `WAITING`，没有会怎样？（八股：状态机设计；场景：IO 等待协程被误重入队）
WAITING 是把“主动让出 CPU（READY）”和“等待外部事件（WAITING）”分开，避免 IO 等待协程被当成可运行任务反复调度。没有它，协程会被误重入队，出现空转、重复恢复、时序错乱。
术语解释：状态机=合法状态和迁移规则；误重入队=本不该再跑的任务被再次放回可执行队列。
2. `CoroutineScheduler::resume` 里 `pending_resumes_` 解决了什么竞态？（八股：并发状态机；场景：恢复请求先于挂起）
pending_resumes_ 解决的是“恢复请求先到，协程还没从 RUNNING 进入 WAITING”的竞态。做法是先记账，等 execute_coroutine 看到 WAITING 时补发一次恢复，避免恢复信号丢失。
术语解释：竞态=并发下先后顺序不确定；补发恢复=延迟执行之前记录的恢复请求。
3. `local_fast_queue/local_queue/global_queue` 三队列的优先级和设计目标分别是什么？（八股：多级队列调度；场景：尾延迟抖动）
优先级是 local_fast_queue > global_queue > local_queue > steal。local_fast_queue 给 IO 恢复低延迟，local_queue 给 yield 续跑保证局部性，global_queue 承接外部提交，窃取用于负载均衡。
术语解释：多级队列调度=不同来源任务走不同队列和优先级；尾延迟=最慢那批请求的延迟（如 p99）。
4. `started_coroutines_` 与工作窃取有何关系？（八股：窃取策略；场景：新任务被窃取造成局部缓存失效）
started_coroutines_ 用来限制工作窃取：只偷“未执行过”的任务，已执行过的协程尽量保持线程亲和。这样能减少跨线程迁移导致的局部缓存失效和恢复复杂度。
术语解释：工作窃取=空闲 worker 从忙 worker 拿任务；线程亲和=任务尽量回到之前运行过的线程。
5. `wait_idle` 为什么用 `pending==0 && active==0` 判空，而不是只看队列空？（八股：并发判空语义；场景：任务在执行但队列空）
面试回答：只看“队列空”会误判，因为任务可能正在执行中。pending==0 && active==0 才表示“既没有排队任务，也没有在途执行任务”。
术语解释：并发判空语义=在并发系统里定义“真正空闲”的标准；active=当前正在跑的任务数。
6. `timed_lock` 采集 `state_lock_wait_ns` 的工程意义是什么？（八股：锁竞争分析；场景：QPS 上不去但 CPU 低）
Hook 路径通过“保存并恢复 errno + 精确映射错误码”保证兼容：EINTR 重试，非 EAGAIN 原样返回，超时映射 ETIMEDOUT，缺符号返回 ENOSYS。另外 ScopedNonBlocking 在恢复 fd flag 时也保留原 errno。
术语解释：errno 兼容=业务依赖错误码分支时行为不变；错误映射=把内部失败原因转成约定错误码。


7. `IOManager::trigger_events` 为何采用“锁内解绑，锁外执行”？（八股：临界区设计；场景：死锁/长临界区）
trigger_events 先在锁内完成解绑和 epoll 更新，再在锁外调度回调。这样保证状态一致，又避免回调执行太慢把锁占住引发长临界区或死锁。
术语解释：锁内解绑=先把共享状态改对；锁外执行=耗时逻辑不在锁里跑。
8. `TimerManager` 为什么取消用延迟删除，不做堆中间删除？（八股：堆结构复杂度；场景：高频取消导致性能掉）
Timer 取消用“标记取消 + 堆顶清理（lazy delete）”，因为堆中间删除需要额外索引维护，复杂且高频取消成本高。延迟删除把复杂度转到自然出堆时处理，吞吐更稳。
术语解释：延迟删除=先打取消标记，后续再真正回收；堆中间删除=在优先队列中直接删除非堆顶元素。
9. `eventfd` 在 IO+Timer 融合中的必要性是什么？（八股：事件唤醒机制；场景：新增更早 timer 不生效）
eventfd 是唤醒 epoll_wait 的标准通道，特别用于“新增更早 timer / 取消 timer”后立刻重算等待时间。没有它，事件循环可能睡到旧超时点，导致新定时器不及时生效。
术语解释：事件唤醒机制=主动打断阻塞等待让循环立即推进；epoll_wait=内核事件等待调用。
10. Hook `read/write/recv/send` 为什么先临时设非阻塞再循环重试？（八股：非阻塞 IO 语义；场景：EAGAIN 误处理）
Hook 先临时设非阻塞，再在 EAGAIN/EWOULDBLOCK 时挂事件并 yield_waiting，事件到来后重试。这样既保持“看起来像阻塞 API”的用法，又不阻塞线程。
术语解释：非阻塞语义=调用不会长期卡线程；循环重试=等待条件满足后再次执行同一 syscall。
11. Hook 路径如何保证 `errno` 与原生语义一致？（八股：系统调用兼容；场景：业务依赖 errno 分支）
Hook 路径通过“保存并恢复 errno + 精确映射错误码”保证兼容：EINTR 重试，非 EAGAIN 原样返回，超时映射 ETIMEDOUT，缺符号返回 ENOSYS。另外 ScopedNonBlocking 在恢复 fd flag 时也保留原 errno。
术语解释：errno 兼容=业务依赖错误码分支时行为不变；错误映射=把内部失败原因转成约定错误码。


12. `yield_current_waiting` 在普通线程调用为什么要失败而不是 no-op？（八股：接口契约；场景：误调用被静默吞掉）
yield_current_waiting 在普通线程必须失败，因为这是“只能在协程上下文调用”的强契约。做成 no-op 会把 bug 静默吞掉，最终变成更难排查的超时或假死。
术语解释：Fail-fast=尽早明确报错；接口契约=调用方必须满足的前置条件。
