## 4. W3 调度器设计说明

### 4.1 对外接口（CoroutineScheduler）
新增/扩展接口：
1. `start(worker_threads)`：启动 worker 线程池，进入 M:N 调度模式。
2. `stop()`：等待任务清空后停止调度并回收线程。
3. `wait_idle()`：阻塞直到 `pending == 0 && active == 0`。
4. `pending_count()/worker_count()/idle_switch_count()/running()`：运行态观测。
5. `schedule()`：提交协程入口（要求线程池已启动）。
6. `alive_count()/completed_count()/recycle_terminated()`：生命周期统计与回收能力保留。

### 4.2 关键数据结构
1. `workers_`：每个 worker 持有独立 `queue + mutex + cv + thread`。
2. `coroutines_`：调度器拥有的协程对象表（`id -> Coroutine`）。
3. 原子计数：
- `pending_tasks_`：已入队但未开始执行的任务数。
- `active_tasks_`：正在 `resume()` 运行中的任务数。
- `idle_switches_`：idle 协程执行次数（用于验证 idle 机制真实生效）。

### 4.3 并发控制策略
1. 队列修改使用 `worker.mutex` 或 `state_mutex_` 保护。
2. 空闲等待通过 `idle_cv_` + 原子计数达成，避免忙等。
3. 协程状态读写使用原子：`CoroutineState state_` 从普通枚举字段改为 `std::atomic<CoroutineState>`，避免多线程下状态读取数据竞争。

## 5. 运行逻辑（时序）

### 5.1 提交任务（schedule）
1. 分配协程 `id`。
2. 构造协程对象并写入 `coroutines_`。
3. `pending_tasks_++`。
4. 线程池已启动时，按轮询将任务投递到某个 worker 队列。
5. 若线程池未启动，直接拒绝提交并抛出逻辑错误。

### 5.2 Worker 执行循环
1. worker 从自身队列取任务。
2. 队列为空时，执行 `idle_coroutine`：
- `sleep_for(1ms)` 降低空转 CPU。
- `Coroutine::yield_current()` 让出执行权。
- `idle_switches_++` 计数。
3. 取到任务后执行 `execute_coroutine()`：
- `active_tasks_++`。
- 调用 `coroutine->resume()`。
- 若协程 `READY`：重新入队。
- 若协程 `TERM`：回收并累计 `completed_count_`。
- `active_tasks_--`。
4. 每次状态变化后触发 `notify_idle_waiters()`，用于唤醒 `wait_idle()`。

### 5.3 停止与回收（stop）
1. 先 `wait_idle()`，确保没有在途任务。
2. 设置 `stop_requested_ = true`，通知所有 worker。
3. `join` 全部 worker 线程。
4. 清空 worker 容器，复位分发游标。
5. `running() == false`，`worker_count() == 0`。

## 7. 新增测试说明（scheduler_mn_test）

### 7.1 覆盖目标
1. 多线程压力：`4` 个 producer 并发提交协程。
2. 队列并发正确性：校验 `finished`、`yields` 与理论值一致。
3. 无死锁：`wait_idle()` 使用超时守卫（stress 阶段 `15s`，restart 阶段 `10s`）。
4. idle 协程生效：校验 `idle_switch_count() > 0`。
5. 生命周期回收：`pending_count == 0`、`alive_count == 0`。
6. 停止与线程回收：`stop()` 后 `running() == false` 且 `worker_count() == 0`。
7. restart 连续性：`start -> stop -> start -> stop` 再次可用。

### 7.2 验收判定
满足以下条件即判定 W3 通过：
1. 压测阶段所有协程完成，yield 总数精确匹配。
2. 两阶段 `wait_idle_with_timeout` 均在超时前返回。
3. 停止后无遗留协程、无遗留 pending、线程可回收。

1. M:N 调度把“协程并发度”与“线程数量”解耦，提升了多核资源利用率。

### 12.2 高频问答
1. 问：为什么 W3 要先补齐生命周期，再做更多功能。  
答：没有可靠 `start/stop/wait_idle`，后续 IO/Hook 功能难以稳定测试和排障。
2. 问：`wait_idle()` 为什么需要超时版本。  
答：并发系统可能因 bug 卡住，超时守卫可避免测试无限等待。
3. 问：为什么 W3 强调“恢复后可 restart”。  
答：这决定运行时是否支持多轮压测、回归与服务热重启验证。
