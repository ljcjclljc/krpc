## 4. 核心设计说明

### 4.1 KrpcChannel：Protobuf 编解码 + Sync/Async
`KrpcChannel` 作为 RPC 调用中的编解码与传输封装层，建议具备：
1. `invoke(...)`：同步调用接口。
2. `invoke_async(...)`：异步调用接口（返回 future 或等价异步句柄）。
3. Protobuf 编解码能力：
- 请求帧编码：`service/method/payload/metadata/timeout/retries`。
- 响应帧解码：`code/message/payload/retryable/effective_timeout/cancelled`。
- 使用 `proto/krpc.proto` + `protoc` 生成 `krpc.pb.h/krpc.pb.cc`，通道层调用
  `SerializeToString/ParseFromString`，避免手写 wire 逻辑。
4. 兼容策略：
- 默认保留 Raw 模式，保证与既有调用链兼容。
- 通过 header/metadata（如 `x-krpc-codec=protobuf`）切换 Protobuf 模式。

### 4.2 超时与取消语义
建议统一语义如下：
1. `deadline_exceeded`：上游预算耗尽，返回超时错误（如 504）。
2. `cancelled`：请求被主动取消，返回取消错误（可与超时区分）。
3. 生效超时预算：
- `effective_timeout_ms = min(上游剩余预算, 下游配置)`。
- 下游配置为 0 时，表示不额外限制，仅受上游预算约束。
4. 取消优先级：
- 在发起下游调用前先检查上下文取消状态。
- 重试前再次检查取消状态，避免取消后继续放大请求。

### 4.3 重试预算接入调用链
建议接入点：
1. 首发请求前：`record_request()`。
2. 每次重试前：`try_acquire_retry_token()`。
3. 预算耗尽：返回 `retry_budget_exhausted`（如 429），停止继续重试。

建议可观测字段：
1. `attempts`：本次总尝试次数。
2. `effective_timeout_ms`：本次生效超时。
3. `retry_budget_request_count`：窗口内请求数。
4. `retry_budget_retry_count`：窗口内重试数。
5. `retry_budget_max_tokens/available_tokens`：预算上限与剩余量。
6. `cancelled/cancel_reason`：取消状态与原因。

## 6. 使用示例（建议）

### 6.1 同步调用（Raw）
```cpp
rpc::client::RpcRequest req;
req.service = "gateway.backend";
req.method = "Echo";
req.payload = "hello";
req.timeout_ms = 300;
req.max_retries = 1;

auto resp = rpc::client::default_client()->invoke(req);
```

### 6.2 同步调用（Protobuf）
```cpp
rpc::client::RpcRequest req;
req.service = "gateway.backend";
req.method = "Echo";
req.payload = "hello";
req.timeout_ms = 300;
req.max_retries = 1;
req.metadata["x-krpc-codec"] = "protobuf";

auto resp = rpc::client::default_client()->invoke(req);
```

## 7. 测试与验收建议

### 7.1 建议新增测试覆盖
1. Protobuf 请求/响应编解码正确。
2. `invoke` 与 `invoke_async` 返回语义一致。
3. deadline 超时与主动取消语义可区分。
4. 重试预算耗尽后不再继续重试。
5. 观测字段（attempts/effective_timeout/预算快照）值正确。

满足以下条件可判定 W11 通过：
1. 基础 RPC 调用在 Raw/Protobuf 模式下均可完成。

## 8. 常见问题
1. Protobuf 模式调用失败：
- 检查上下游是否都按 Krpc 帧协议编解码。
2. 明明设置了超时但未生效：
- 检查是否有上游 deadline 覆盖，以及 `effective_timeout_ms` 实际值。
3. 重试次数异常：
- 同时检查 `max_retries` 与预算 token 是否命中限制。

## 9. 结论
W11 的核心价值不是新增单一功能，而是把“通道编解码能力 + 超时取消治理 + 重试预算治理”统一到同一 RPC 调用语义中，并给出可配置、可观测、可回归的工程闭环。
