## 4. 整体设计概览

W6 采用“**按线程开关 Hook + 协程内按需异步等待**”模式：

1. Hook 默认关闭（TLS `false`）。
2. 只有在下面三个条件同时满足时，才走协程化路径：
- 当前线程 Hook 开关开启。
- 当前线程存在运行中的协程（`Coroutine::current() != nullptr`）。
- 已绑定可用的 `IOManager`。
3. 其余情况全部走原生系统调用，不改变线程原有行为。

这样可保证：
1. 非 Hook 线程不受影响。
2. 非协程上下文（普通线程函数）不被强制协程化。
3. IO 等待能力复用现有 `IOManager`/`epoll`/调度器链路，无需引入新事件循环。

## 5. 运行逻辑详解（调用链）

### 5.1 Hook 开关与运行时绑定
关键接口在 `hook.h`：
1. `set_hook_enabled(bool)` / `hook_enabled()`：线程本地开关（TLS）。
2. `ScopedHookEnable`：作用域开关，构造时设置、析构时恢复。
3. `set_hook_io_manager(IOManager*)` / `hook_io_manager()`：绑定全局可用的 `IOManager`。

实现位置：`src/runtime/hook.cpp`。

---

### 5.2 `sleep` Hook 路径
函数：`extern "C" unsigned int sleep(unsigned int seconds)`。

#### 路径 A：不满足 Hook 条件
直接调用真实 `sleep`（通过 `dlsym(RTLD_NEXT, "sleep")` 获取）。

#### 路径 B：满足 Hook 条件
1. 通过 `IOManager::add_timer()` 注册一个 `seconds` 后触发的回调。
2. 回调只负责设置 `done=true`。
3. 当前协程循环 `yield_current()`，直到 `done=true`。
4. 返回 `0`（表示睡眠完成）。

核心意义：
1. 当前 worker 线程不会被 `sleep` 真正阻塞。
2. 其他协程可继续执行，体现“阻塞调用协程化”。

---

### 5.3 `read/write/recv/send` Hook 路径
核心模板函数：`run_io_hooked(fd, wait_event, syscall_once)`。

#### 步骤 1：判定是否走 Hook
不满足 Hook 条件则直接执行原生调用 `syscall_once()` 并返回。

#### 步骤 2：临时切换非阻塞
进入 Hook 后使用 `ScopedNonBlocking`：
1. `F_GETFL` 读取原 flags。
2. 如果不是非阻塞，临时加上 `O_NONBLOCK`。
3. 退出作用域时恢复原 flags（并保护 `errno` 不被恢复动作污染）。

#### 步骤 3：重试循环
1. 先执行一次真实系统调用。
2. 若成功（`>=0`）直接返回。
3. 若 `EINTR`，继续重试。
4. 若 `EAGAIN/EWOULDBLOCK`，进入事件等待。
5. 其余错误原样返回，并保留原 `errno`。

#### 步骤 4：等待 fd 事件
`wait_fd_event_once(fd, event)`：
1. 调用 `io_manager->add_event(fd, event, callback)` 注册一次性回调。
2. 回调仅设置 `notified=true`。
3. 当前协程循环 `yield_current()`，等待 `notified`。
4. 通知后返回到重试循环，再次执行系统调用。

---

### 5.4 `epoll` 回调如何接上 Hook
本仓库当前 `IOManager` 采用“**只回调，不恢复旧协程**”模型：

1. `add_event(fd, event, callback)` 把回调存入 `FdContext::EventContext`。
2. `event_loop()` 的 `epoll_wait` 收到事件后调用 `FdContext::trigger_events(...)`。
3. `trigger_events` 取出对应回调并调用 `scheduler.schedule(callback)`。
4. 回调在调度器的新协程中执行，置位 `notified=true`。
5. 原等待协程在下一次 `yield` 返回后继续重试读写。

也就是说，Hook 不是“唤醒旧协程”，而是“通过事件回调改变共享标志 + 旧协程主动让出并轮询标志”。

## 6. 源码关键函数解读

### 6.1 `src/runtime/hook.cpp`
1. `hook_path_enabled()`  
作用：统一判定 Hook 生效条件（TLS 开关 + 当前协程 + IOManager 绑定）。

2. `ScopedNonBlocking`  
作用：临时把 fd 设为非阻塞，析构时恢复原状态，同时保护 `errno`。

3. `wait_fd_event_once()`  
作用：在 `IOManager` 中注册一次读/写事件回调，当前协程 `yield` 等待通知。

4. `run_io_hooked()`  
作用：封装 `read/write/recv/send` 的统一重试框架，处理 `EINTR`、`EAGAIN`、事件等待。

5. `sleep/read/write/recv/send`（`extern "C"`）  
作用：导出与 libc 同名符号，在同进程内提供 Hook 行为分流。

6. `send` 的 `EPERM` 兼容分支  
逻辑：当 `send(..., flags=0)` 返回 `EPERM` 时，回退到 `write` 实现，增强兼容性。

---

### 6.2 `src/runtime/io_manager.cpp`
1. `add_event_internal()`  
只允许绑定回调，不再接受协程 ID。

2. `event_loop()`  
`epoll_wait` + timer 超时融合；触发事件后转到 `trigger_events`。

3. `FdContext::trigger_events()`  
锁内解绑和更新 `epoll` 掩码，锁外 `scheduler.schedule(callback)`。

这套实现是 W6 Hook 能够复用的关键基础设施。

## 7. `errno` 兼容与边界行为

### 7.1 已对齐语义
1. 非 Hook 路径：直接透传原生系统调用返回值和 `errno`。
2. Hook 路径下普通错误：保留系统调用原始 `errno` 返回。
3. `EINTR`：重试。
4. `EAGAIN/EWOULDBLOCK`：注册事件 + yield，事件到达后重试。
5. 非法 fd（如 `read(-1, ...)`）：保持 `-1 + EBADF`。

### 7.2 当前边界策略
1. 若事件注册失败（`add_event` 返回 `false`），当前实现返回 `-1` 且置 `errno=EBUSY`。
2. `send(flags=0)` 且 `EPERM` 时回退 `write`，`flags!=0` 时不回退。

## 9. 与 W4/W5 的关系
1. W4 提供了 `epoll + FdContext + callback` 事件分发框架。
2. W5 提供了 `TimerManager + IOManager::add_timer` 能力。
3. W6 在此基础上把阻塞 syscall 统一接入“事件回调 + 协程 yield 重试”模型。

即：W6 不是新起炉灶，而是把 W4/W5 运行时能力串成可直接使用的 Hook 首批闭环。
