# 第二部分：基础知识深度讲解

## 2.1 并发编程基础

### 2.1.1 什么是并发？

**核心概念**：并发是指两个或多个任务在**同一时间段内**交替执行，而不是同时执行。

```
并发 vs 并行：

并发 (Concurrency):
任务 A ────┐          ├── 任务 B ────
          └── 交替执行 ─┘

并行 (Parallelism):
任务 A ───────────────
          同时执行
任务 B ───────────────

区别：
- 并发：逻辑上的同时，物理上的交替
- 并行：物理上的同时，需要多核 CPU 支持
```

### 2.1.2 并发模型演进

```
第一代：多进程 (Process-per-Connection)
┌────────┐ ┌────────┐ ┌────────┐
│Process1│ │Process2│ │Process3│
│ 100MB  │ │ 100MB  │ │ 100MB  │
└────────┘ └────────┘ └────────┘
优点：隔离性好
缺点：内存占用大，创建/销毁开销大

第二代：多线程 (Thread-per-Connection)
┌─────────┐
│ Process │
│ ┌─────┐ │
│ │T1 2MB│ │
│ │T2 2MB│ │
│ │T3 2MB│ │
│ └─────┘ │
└─────────┘
优点：轻量级，共享内存
缺点：上下文切换开销大，编程复杂

第三代：协程 (Coroutine)
┌─────────┐
│ Thread  │
│ ┌─────┐ │
│ │C1 2KB│ │
│ │C2 2KB│ │
│ │C3 2KB│ │
│ └─────┘ │
└─────────┘
优点：超轻量，用户态调度
缺点：需要调度器支持

第四代：协程 + Reactor (本项目)
┌─────────────────────────────┐
│     IOManager (4 Threads)    │
│  ┌───────────────────────┐  │
│  │ Epoll + 协程池         │  │
│  │ 10,000+ 协程          │  │
│  └───────────────────────┘  │
└─────────────────────────────┘
优点：高并发，低资源，易编程
缺点：实现复杂
```

### 2.1.3 并发中的核心问题

#### 1. 竞态条件 (Race Condition)

```cpp
// 错误示例：多线程共享变量
int counter = 0;

void increment() {
    counter++;  // 非原子操作，存在竞态
}

// counter++ 的实际操作：
// 1. 读取 counter 到寄存器
// 2. 寄存器 +1
// 3. 写回 counter
// 如果三个线程同时执行，结果可能不是 3

// 解决方案：使用原子操作
std::atomic<int> counter{0};
counter++;  // 原子操作，线程安全
```

#### 2. 死锁 (Deadlock)

```cpp
// 死锁示例
std::mutex mutex1;
std::mutex mutex2;

void thread1() {
    std::lock_guard<std::mutex> lock1(mutex1);
    std::this_thread::sleep_for(std::chrono::milliseconds(100));
    std::lock_guard<std::mutex> lock2(mutex2);  // 等待 mutex2
}

void thread2() {
    std::lock_guard<std::mutex> lock2(mutex2);
    std::this_thread::sleep_for(std::chrono::milliseconds(100));
    std::lock_guard<std::mutex> lock1(mutex1);  // 等待 mutex1
}

// thread1 持有 mutex1，等待 mutex2
// thread2 持有 mutex2，等待 mutex1
// 互相等待，死锁！

// 解决方案：统一加锁顺序
void thread1() {
    std::lock(mutex1, mutex2);  // 同时加锁，避免死锁
    std::lock_guard<std::mutex> lock1(mutex1, std::adopt_lock);
    std::lock_guard<std::mutex> lock2(mutex2, std::adopt_lock);
}

void thread2() {
    std::lock(mutex1, mutex2);  // 相同的加锁顺序
    std::lock_guard<std::mutex> lock1(mutex1, std::adopt_lock);
    std::lock_guard<std::mutex> lock2(mutex2, std::adopt_lock);
}
```

#### 3. 上下文切换开销

```
上下文切换的成本：

进程切换：
- 切换模式：用户态 → 内核态
- 保存内容：进程控制块 (PCB)、页表、文件描述符等
- 开销：~1-5 微秒
- 缓存影响：TLB、CPU 缓存失效

线程切换：
- 切换模式：用户态 → 内核态
- 保存内容：寄存器、栈指针、程序计数器等
- 开销：~1-2 微秒
- 缓存影响：CPU 缓存可能失效

协程切换：
- 切换模式：纯用户态
- 保存内容：寄存器、栈指针
- 开销：~50-100 纳秒
- 缓存影响：TLB、CPU 缓存保持

性能对比：
协程切换比线程切换快 10-20 倍！
```

### 2.1.4 项目中的并发设计

**本项目采用 M:N 协程调度模型：**

```cpp
// [scheduler.cpp:66-147] 调度器核心逻辑
void Scheduler::run() {
    // 每个线程独立的调度循环
    while(true) {
        // 1. 从任务队列取任务
        {
            std::lock_guard<std::mutex> lock(m_mutex);
            auto it = m_tasks.begin();
            while(it != m_tasks.end()) {
                // 跳过指定给其他线程的任务
                if(it->thread != -1 && it->thread != thread_id) {
                    it++;
                    tick_me = true;
                    continue;
                }
                task = *it;
                m_tasks.erase(it);
                m_activeThreadCount++;
                break;
            }
        }
        
        // 2. 执行任务（恢复协程）
        if(task.fiber) {
            task.fiber->resume();
            m_activeThreadCount--;
        }
        
        // 3. 无任务时执行空闲协程
        else {
            m_idleThreadCount++;
            idle_fiber->resume();
            m_idleThreadCount--;
        }
    }
}
```

**设计亮点：**

1. **线程独立调度**：每个线程有自己的任务队列，减少锁竞争
2. **任务窃取**：通过 `tick_me` 通知其他线程取任务
3. **协程复用**：协程执行完后可以 reset 复用，减少内存分配

## 2.2 ucontext 深度解析

### 2.2.1 什么是 ucontext？

ucontext 是 **Linux 提供的用户态上下文切换机制**，允许程序在用户态保存和恢复执行上下文。

```
上下文 (Context) = CPU 寄存器状态 + 栈指针 + 信号掩码

┌─────────────────────────────────────────────────────┐
│                  CPU 寄存器                          │
│  • RIP: 指令指针 (程序计数器)                       │
│  • RSP: 栈指针                                      │
│  • RBP: 基址指针                                    │
│  • RBX, R12-R15: 通用寄存器                         │
│  • ...                                             │
├─────────────────────────────────────────────────────┤
│                  栈信息                              │
│  • ss_sp: 栈起始地址                                │
│  • ss_size: 栈大小                                  │
│  • uc_link: 下一个执行的上下文                      │
├─────────────────────────────────────────────────────┤
│                  信号掩码                            │
│  • uc_sigmask: 阻塞的信号集合                       │
└─────────────────────────────────────────────────────┘
```

### 2.2.2 ucontext 族函数详解

#### 1. getcontext - 获取当前上下文

```c
#include <ucontext.h>

int getcontext(ucontext_t *ucp);

// 功能：将当前执行上下文保存到 ucp 中
// 返回值：成功返回 0，失败返回 -1

// 使用示例：
ucontext_t main_ctx, child_ctx;

// 保存主上下文
getcontext(&main_ctx);

// 此时 main_ctx 包含了当前的执行状态
// 包括：寄存器、栈指针、程序计数器等
```

**底层实现原理：**

```assembly
# getcontext 的伪代码（汇编级别）
getcontext:
    # 保存通用寄存器
    mov    %rbx, (%rdi)          # RBX
    mov    %rbp, 8(%rdi)         # RBP
    mov    %r12, 16(%rdi)        # R12
    mov    %r13, 24(%rdi)        # R13
    mov    %r14, 32(%rdi)        # R14
    mov    %r15, 40(%rdi)        # R15
    mov    %rsp, 48(%rdi)        # RSP (栈指针)
    mov    (%rsp), %rax          # 读取返回地址
    mov    %rax, 56(%rdi)        # RIP (程序计数器)
    
    # 保存信号掩码
    # ...
    
    xor    %eax, %eax            # 返回 0
    ret
```

#### 2. makecontext - 初始化新上下文

```c
void makecontext(ucontext_t *ucp, void (*func)(), int argc, ...);

// 功能：初始化一个新上下文，设置入口函数和参数
// 参数：
//   ucp: 要初始化的上下文
//   func: 入口函数
//   argc: 参数个数
//   ...: 参数列表

// 使用示例：
ucontext_t ctx;
getcontext(&ctx);  // 先获取当前上下文

// 设置栈
char stack[128000];
ctx.uc_stack.ss_sp = stack;
ctx.uc_stack.ss_size = sizeof(stack);

// 设置入口函数
makecontext(&ctx, my_function, 0);  // my_function 无参数

// 或者带参数
void my_function(int arg1, int arg2);
makecontext(&ctx, my_function, 2, 10, 20);
```

**makecontext 做了什么？**

```c
// makecontext 的内部逻辑（简化版）
void makecontext(ucontext_t *ucp, void (*func)(), int argc, ...) {
    // 1. 设置 uc_link（下一个执行的上下文）
    // 如果当前上下文结束，会跳转到 uc_link
    // 如果 uc_link == NULL，线程退出
    
    // 2. 设置栈指针
    // 将栈指针调整到合适位置，为参数留空间
    
    // 3. 在栈上构造参数
    va_list ap;
    va_start(ap, argc);
    for (int i = 0; i < argc; i++) {
        // 将参数压栈
        void* arg = va_arg(ap, void*);
        // ... 将 arg 放到栈上
    }
    va_end(ap);
    
    // 4. 设置 RIP（指令指针）为 func
    // 这样 swapcontext 后会从 func 开始执行
    ucp->uc_mcontext.gregs[REG_RIP] = (greg_t)func;
}
```

#### 3. swapcontext - 切换上下文

```c
int swapcontext(ucontext_t *oucp, const ucontext_t *ucp);

// 功能：保存当前上下文到 oucp，然后切换到 ucp
// 返回值：成功返回 0，失败返回 -1

// 使用示例：
ucontext_t ctx1, ctx2;

// 初始化 ctx2
getcontext(&ctx2);
ctx2.uc_stack.ss_sp = stack2;
ctx2.uc_stack.ss_size = sizeof(stack2);
makecontext(&ctx2, function2, 0);

// 从 ctx1 切换到 ctx2
// 1. 保存当前状态到 ctx1
// 2. 切换到 ctx2 执行
swapcontext(&ctx1, &ctx2);

// 当 ctx2 调用 swapcontext 切回时，从这里继续
```

**swapcontext 执行流程：**

```
swapcontext(&ctx1, &ctx2) 的执行过程：

┌─────────────────────────────────────────────────────────┐
│  切换前 (当前在 ctx1)                                  │
│  ┌─────────────────────────────────────────────────┐   │
│  │  CPU 寄存器                                      │   │
│  │    RIP = 0x400500 (当前指令)                    │   │
│  │    RSP = 0x7fff0000 (栈顶)                     │   │
│  │    RBP = 0x7fff1000                             │   │
│  │    ...                                           │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
                         │
                         │ swapcontext 调用
                         ▼
┌─────────────────────────────────────────────────────────┐
│  保存阶段                                               │
│  1. 将当前寄存器保存到 ctx1                             │
│     ctx1.rip = 0x400500                                │
│     ctx1.rsp = 0x7fff0000                              │
│     ctx1.rbp = 0x7fff1000                              │
│     ...                                                 │
└─────────────────────────────────────────────────────────┘
                         │
                         │ 恢复 ctx2 的寄存器
                         ▼
┌─────────────────────────────────────────────────────────┐
│  恢复阶段                                               │
│  1. 从 ctx2 恢复寄存器                                  │
│     RIP = ctx2.rip (function2 的地址)                  │
│     RSP = ctx2.rsp (stack2 的栈顶)                     │
│     RBP = ctx2.rbp                                     │
│     ...                                                 │
└─────────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│  切换后 (现在在 ctx2)                                   │
│  • CPU 从 ctx2.rip 开始执行 (function2)                │
│  • 栈切换到 ctx2 的栈                                   │
│  • 仿佛从 function2 开始执行一样                        │
└─────────────────────────────────────────────────────────┘
```

**底层实现原理（汇编）：**

```assembly
# swapcontext 的伪代码（x86-64）
swapcontext:
    # 参数：RDI = oucp, RSI = ucp
    
    # === 保存当前上下文到 oucp ===
    mov    %rbx, (%rdi)           # oucp->uc_mcontext.gregs[REG_RBX]
    mov    %rbp, 8(%rdi)          # oucp->uc_mcontext.gregs[REG_RBP]
    mov    %r12, 16(%rdi)         # R12
    mov    %r13, 24(%rdi)         # R13
    mov    %r14, 32(%rdi)         # R14
    mov    %r15, 40(%rdi)         # R15
    mov    %rsp, 48(%rdi)         # oucp->uc_mcontext.gregs[REG_RSP]
    lea    8(%rsp), %rdx          # 计算返回地址
    mov    %rdx, 56(%rdi)         # oucp->uc_mcontext.gregs[REG_RIP]
    
    # 保存浮点寄存器 (如果需要)
    # ...
    
    # 保存信号掩码
    # ...
    
    # === 恢复 ucp 的上下文 ===
    mov    56(%rsi), %rdx         # ucp->uc_mcontext.gregs[REG_RIP]
    mov    48(%rsi), %rsp         # ucp->uc_mcontext.gregs[REG_RSP]
    mov    (%rsi), %rbx           # RBX
    mov    8(%rsi), %rbp          # RBP
    mov    16(%rsi), %r12         # R12
    mov    24(%rsi), %r13         # R13
    mov    32(%rsi), %r14         # R14
    mov    40(%rsi), %r15         # R15
    
    # 恢复信号掩码
    # ...
    
    # 跳转到新的 RIP
    jmp    *%rdx                  # 跳转到 ucp 的 RIP
```

### 2.2.3 项目中的 ucontext 应用

**协程切换的核心代码：**

```cpp
// [fiber.cpp:84-107] resume 实现
void Fiber::resume() {
    m_state = RUNNING;
    
    if(m_runInScheduler) {
        SetThis(this);
        // 从调度协程切换到当前协程
        // 保存调度协程的上下文到 t_scheduler_fiber->m_ctx
        // 恢复当前协程的上下文从 m_ctx
        swapcontext(&t_scheduler_fiber->m_ctx, &m_ctx);
    } else {
        SetThis(this);
        // 从主协程切换到当前协程
        swapcontext(&t_thread_fiber->m_ctx, &m_ctx);
    }
}

// [fiber.cpp:110-136] yield 实现
void Fiber::yield() {
    if(m_state != TERM) {
        m_state = READY;
    }
    
    if(m_runInScheduler) {
        SetThis(t_scheduler_fiber);
        // 保存当前协程的上下文到 m_ctx
        // 恢复调度协程的上下文从 t_scheduler_fiber->m_ctx
        swapcontext(&m_ctx, &t_scheduler_fiber->m_ctx);
    } else {
        SetThis(t_thread_fiber.get());
        swapcontext(&m_ctx, &t_thread_fiber->m_ctx);
    }
}
```

## 2.3 IO 多路复用深度讲解

### 2.3.1 为什么需要 IO 多路复用？

**传统 IO 模型的问题：**

```cpp
// 传统方式：一个连接一个线程
void handle_connection(int fd) {
    char buf[4096];
    while (true) {
        // 阻塞读取，线程被挂起
        int n = read(fd, buf, sizeof(buf));
        if (n > 0) {
            process_data(buf, n);
        } else {
            break;
        }
    }
}

// 问题：
// 10,000 个连接需要 10,000 个线程
// 每个线程 8MB 栈 = 80GB 内存
// 大量线程上下文切换，CPU 利用率低
```

**IO 多路复用的解决方案：**

```cpp
// IO 多路复用：一个线程监控多个连接
void event_loop() {
    while (true) {
        // 同时监控多个 fd
        int ready = epoll_wait(epfd, events, MAX_EVENTS, timeout);
        
        for (int i = 0; i < ready; i++) {
            int fd = events[i].data.fd;
            if (events[i].events & EPOLLIN) {
                handle_read(fd);
            }
            if (events[i].events & EPOLLOUT) {
                handle_write(fd);
            }
        }
    }
}

// 优势：
// 10,000 个连接只需要 1 个线程（或少数几个）
// 内存占用小，CPU 利用率高
```

### 2.3.2 IO 多路复用模型对比

#### select vs poll vs epoll

| 特性 | select | poll | epoll |
|------|--------|------|-------|
| **数据结构** | bitmap（位图） | 数组 | 红黑树 |
| **最大连接数** | 1024（可修改） | 无限制 | 无限制 |
| **时间复杂度** | O(n) | O(n) | O(1) |
| **内存拷贝** | 每次都拷贝 | 每次都拷贝 | 通过 mmap 共享内存 |
| **效率变化** | 连接越多越慢 | 连接越多越慢 | 不随连接数变化 |

**select 的实现原理：**

```c
// select 使用 bitmap 表示 fd
// 每个 bit 代表一个 fd 是否需要监控

// 假设监控 fd 1, 3, 5
fd_set readfds;
FD_ZERO(&readfds);
FD_SET(1, &readfds);
FD_SET(3, &readfds);
FD_SET(5, &readfds);

// select 的内部结构（简化版）
// bitmap: 000000000000000000000000000101010
//                         ^5      ^3   ^1

// 每次调用 select 都需要：
// 1. 将用户态的 bitmap 拷贝到内核态
// 2. 内核遍历所有 fd（通常 1024 个）
// 3. 检查哪些 fd 就绪
// 4. 修改 bitmap，拷贝回用户态

int max_fd = 5;
int ret = select(max_fd + 1, &readfds, NULL, NULL, NULL);
// 时间复杂度：O(max_fd)
```

**poll 的实现原理：**

```c
// poll 使用数组存储要监控的 fd
struct pollfd fds[3];
fds[0].fd = 1;
fds[0].events = POLLIN;
fds[1].fd = 3;
fds[1].events = POLLIN;
fds[2].fd = 5;
fds[2].events = POLLIN;

// poll 的内部结构
struct pollfd {
    int   fd;         // 文件描述符
    short events;     // 监控的事件
    short revents;    // 返回的事件
};

// 每次调用 poll 都需要：
// 1. 将数组拷贝到内核态
// 2. 内核遍历数组中的所有 fd
// 3. 检查哪些 fd 就绪
// 4. 修改 revents，拷贝回用户态

int ret = poll(fds, 3, -1);
// 时间复杂度：O(n)，n 是监控的 fd 数量
```

**epoll 的实现原理：**

```c
// epoll 使用红黑树存储要监控的 fd
int epfd = epoll_create1(0);

// 添加 fd 到 epoll
struct epoll_event ev;
ev.events = EPOLLIN;
ev.data.fd = 1;
epoll_ctl(epfd, EPOLL_CTL_ADD, 1, &ev);

// epoll 的内部结构（简化版）
struct eventpoll {
    // 红黑树：存储所有监控的 fd
    struct rb_root rbr;
    
    // 就绪链表：存储就绪的 fd
    struct list_head rdllist;
    
    // 就绪事件数组（通过 mmap 共享）
    struct epoll_event *events;
};

// 每次调用 epoll_wait：
// 1. 检查就绪链表是否为空
// 2. 如果为空，挂起当前进程/线程
// 3. 当某个 fd 就绪时，中断处理程序将其加入就绪链表
// 4. 唤醒进程/线程，返回就绪的 fd

struct epoll_event events[MAX_EVENTS];
int ret = epoll_wait(epfd, events, MAX_EVENTS, timeout);
// 时间复杂度：O(1) + O(k)，k 是就绪的 fd 数量
```

### 2.3.3 epoll 深度剖析

#### epoll 的三个核心函数

**1. epoll_create - 创建 epoll 实例**

```c
int epfd = epoll_create1(0);

// 内部做了什么？
// 1. 分配一个 struct eventpoll
// 2. 创建一个匿名文件（file descriptor）
// 3. 初始化红黑树和就绪链表
// 4. 返回 fd

// 内核源码（简化版）:
struct eventpoll *ep;
ep = kmalloc(sizeof(struct eventpoll), GFP_KERNEL);
ep->rbr = RB_ROOT;
init_waitqueue_head(&ep->wq);
ep->events = NULL;

// 创建匿名文件
anon_inode_getfile("[eventpoll]", &eventpoll_fops, ep);
```

**2. epoll_ctl - 添加/修改/删除监控**

```c
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);

// 参数说明：
// epfd: epoll 实例的 fd
// op: 操作类型
//   - EPOLL_CTL_ADD: 添加 fd
//   - EPOLL_CTL_MOD: 修改 fd 的事件
//   - EPOLL_CTL_DEL: 删除 fd
// fd: 要监控的文件描述符
// event: 监控的事件

// 内部做了什么？
// EPOLL_CTL_ADD:
// 1. 从 eventpoll 的红黑树中查找 fd
// 2. 如果不存在，创建一个 struct epitem
// 3. 初始化 epitem，包括 fd、事件等
// 4. 将 epitem 插入红黑树
// 5. 在 fd 的等待队列中注册回调函数（ep_poll_callback）

// 内核源码（简化版）:
struct epitem *epi;
epi = kmem_cache_alloc(epi_cache, GFP_KERNEL);

// 初始化 epitem
epi->epdata.fd = fd;
epi->epdata.event = *event;
epi->epdata.ep = ep;

// 插入红黑树
ep_rbtree_insert(ep, epi);

// 注册回调
f_op->poll(fd, &epq.pt, key);
// 这个 poll 会注册 ep_poll_callback
```

**3. epoll_wait - 等待事件**

```c
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);

// 参数说明：
// epfd: epoll 实例的 fd
// events: 输出参数，存储就绪的事件
// maxevents: 最多返回多少个事件
// timeout: 超时时间（毫秒）

// 内部做了什么？
// 1. 检查就绪链表是否为空
// 2. 如果不为空，直接返回就绪的事件
// 3. 如果为空，进入休眠（sleep）
//    - 将当前进程/线程加入等待队列
//    - 释放 CPU，调度其他进程/线程
// 4. 当某个 fd 就绪时：
//    - 中断处理程序调用 ep_poll_callback
//    - ep_poll_callback 将对应的 epitem 加入就绪链表
//    - 唤醒等待队列中的进程/线程
// 5. 进程/线程被唤醒，从就绪链表取事件

// 内核源码（简化版）:
ep_poll_callback(struct epitem *epi) {
    // 将 epitem 加入就绪链表
    list_add_tail(&epi->rdllink, &ep->rdllist);
    
    // 唤醒等待队列
    if (waitqueue_active(&ep->wq)) {
        wake_up_locked(&ep->wq);
    }
}

ep_poll(struct eventpoll *ep) {
    while (!ep->rdllist.empty()) {
        // 从就绪链表取事件
        struct epitem *epi = list_first_entry(&ep->rdllist);
        // 拷贝到用户空间
        copy_to_user(events, &epi->epdata.event);
    }
    
    // 如果没有就绪事件，进入休眠
    set_current_state(TASK_INTERRUPTIBLE);
    schedule();
}
```

### 2.3.4 LT vs ET 深度对比

**Level Triggered (水平触发)：**

```
特点：
- 只要缓冲区有数据，就一直通知
- 编程简单，不容易遗漏事件
- 可能会频繁唤醒

工作流程：
┌─────────────────────────────────────────────────────┐
│ 缓冲区状态变化：                                    │
│  空 → 有数据 → 通知 → read(100) → 剩 100 字节       │
│   → 再次通知 → read(100) → 剩 0 字节               │
│   → 空                                            │
└─────────────────────────────────────────────────────┘

代码示例：
// LT 模式（默认）
epoll_event ev;
ev.events = EPOLLIN;  // 默认就是 LT
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);

// 处理
epoll_wait(epfd, events, MAX_EVENTS, -1);
for (int i = 0; i < nfds; i++) {
    int fd = events[i].data.fd;
    // 可以分次读取，不必一次性读完
    int n = read(fd, buf, 100);
    // 即使没读完，下次 epoll_wait 还会通知
}
```

**Edge Triggered (边缘触发)：**

```
特点：
- 只在状态变化时通知一次
- 编程复杂，需要一次性读完
- 减少唤醒次数，性能更高

工作流程：
┌─────────────────────────────────────────────────────┐
│ 缓冲区状态变化：                                    │
│  空 → 有数据 → 通知一次                             │
│   → read(100) → 剩 100 字节（不会再次通知）         │
│   → 必须循环读取直到 EAGAIN                         │
│   → 空                                            │
└─────────────────────────────────────────────────────┘

代码示例：
// ET 模式
epoll_event ev;
ev.events = EPOLLIN | EPOLLET;  // 添加 EPOLLET 标志
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);

// 设置为非阻塞
int flags = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);

// 处理（必须循环读取）
epoll_wait(epfd, events, MAX_EVENTS, -1);
for (int i = 0; i < nfds; i++) {
    int fd = events[i].data.fd;
    
    while (true) {
        int n = read(fd, buf, sizeof(buf));
        if (n > 0) {
            process_data(buf, n);
        } else if (n == 0) {
            // 连接关闭
            close(fd);
            break;
        } else {
            if (errno == EAGAIN) {
                // 数据读完了
                break;
            } else {
                // 真正的错误
                handle_error();
                break;
            }
        }
    }
}
```

**对比总结：**

| 特性 | LT | ET |
|------|----|----|
| 通知时机 | 有数据就一直通知 | 状态变化时通知一次 |
| 编程复杂度 | 简单 | 复杂（必须循环读） |
| 性能 | 较低（频繁唤醒） | 较高（减少唤醒） |
| 适用场景 | 一般场景 | 高性能场景 |

### 2.3.5 项目中的 epoll 应用

**IOManager 的 epoll 实现：**

```cpp
// [ioscheduler.cpp:296-386] idle 实现
void IOManager::idle() {
    const uint64_t MAX_EVENTS = 256;
    std::unique_ptr<epoll_event[]> events(new epoll_event[MAX_EVENTS]);
    
    while(true) {
        if(stopping()) break;
        
        // 1. 计算超时时间
        uint64_t nextTimer = getNextTimer();
        uint64_t timeout = std::min(5000ull, nextTimer);
        
        // 2. epoll_wait
        int rt = epoll_wait(m_epfd, events.get(), MAX_EVENTS, timeout);
        
        // 3. 处理定时器
        std::vector<std::function<void()>> cbs;
        listExpiredCb(cbs);
        for(const auto& cb : cbs) {
            scheduleLock(cb);
        }
        
        // 4. 处理 IO 事件
        for(int i = 0; i < rt; ++i) {
            epoll_event& event = events[i];
            
            // 处理 tickle fd
            if(event.data.fd == m_tickleFds[0]) {
                uint8_t dummy[256];
                while(read(m_tickleFds[0], dummy, sizeof(dummy)) > 0);
                continue;
            }
            
            // 获取 FdContext
            FdContext* fd_ctx = (FdContext*)event.data.ptr;
            
            // 处理错误事件
            if(event.events & (EPOLLERR | EPOLLHUP)) {
                event.events |= (EPOLLIN | EPOLLOUT);
            }
            
            // 提取真实事件
            int real_events = NONE;
            if(event.events & EPOLLIN) real_events |= READ;
            if(event.events & EPOLLOUT) real_events |= WRITE;
            
            // 从 epoll 中删除事件（ET 模式）
            int left_events = (fd_ctx->events & ~real_events);
            int op = left_events ? EPOLL_CTL_MOD : EPOLL_CTL_DEL;
            epoll_ctl(m_epfd, op, fd_ctx->fd, &event);
            
            // 触发回调或恢复协程
            if(real_events & READ) fd_ctx->triggerEvent(READ);
            if(real_events & WRITE) fd_ctx->triggerEvent(WRITE);
        }
        
        Fiber::GetThis()->yield();
    }
}
```

**设计亮点：**

1. **ET 模式**：使用边缘触发，减少系统调用
2. **Re-arm 机制**：每次处理完事件后重新注册
3. **定时器结合**：epoll_wait 超时设置为下一个定时器时间
4. **tickle 唤醒**：通过 pipe 唤醒阻塞的 epoll_wait

## 2.4 Reactor 模式深度讲解

### 2.4.1 什么是 Reactor？

Reactor 是一种**事件驱动的并发模式**，核心思想是将 IO 事件的检测和处理分离。

**传统模式 vs Reactor 模式：**

```
传统模式 (阻塞):
Thread 1:  read() ━━━━━━━━━━ 等待数据 ━━━━━━━━━━
Thread 2:  read() ━━━━━━━━━━ 等待数据 ━━━━━━━━━━
Thread 3:  read() ━━━━━━━━━━ 等待数据 ━━━━━━━━━━

Reactor 模式 (非阻塞):
Main Thread:  epoll_wait() ━━━ 返回 (fd1 可读)
            ├─ handle_fd1_read()
            ├─ epoll_wait() ━━━ 返回 (fd2 可读)
            ├─ handle_fd2_read()
            └─ ...
```

### 2.4.2 Reactor 的核心组件

```
┌─────────────────────────────────────────────────────────┐
│                    Reactor 模式                           │

### 2.4.4 项目中的 Reactor 实现

**本项目采用：Reactor + 协程**

```
┌─────────────────────────────────────────────────────┐
│              IOManager (4 个线程)                    │
│  每个线程都是一个独立的 Reactor                     │
└─────────────────────────────────────────────────────┘
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│  Thread 1   │ │  Thread 2   │ │  Thread 3   │
│ ┌─────────┐ │ │ ┌─────────┐ │ │ ┌─────────┐ │
│ │Reactor  │ │ │ │Reactor  │ │ │ │Reactor  │ │
│ │+ 协程池 │ │ │ │+ 协程池 │ │ │ │+ 协程池 │ │
│ └─────────┘ │ │ └─────────┘ │ │ └─────────┘ │
└─────────────┘ └─────────────┘ └─────────────┘

每个线程：
1. epoll_wait() 等待事件
2. 事件到来 → 恢复对应协程
3. 协程执行 → yield() 挂起
4. 回到 epoll_wait()
```

**代码实现：**

```cpp
// [ioscheduler.cpp:296-386] idle = Reactor 的事件循环
void IOManager::idle() {
    std::unique_ptr<epoll_event[]> events(new epoll_event[MAX_EVENTS]);
    
    while(true) {
        // 1. epoll_wait = Event Loop
        int rt = epoll_wait(m_epfd, events.get(), MAX_EVENTS, timeout);
        
        // 2. 处理定时器
        listExpiredCb(cbs);
        for(const auto& cb : cbs) {
            scheduleLock(cb);  // 调度定时器回调
        }
        
        // 3. 处理 IO 事件
        for(int i = 0; i < rt; ++i) {
            FdContext* fd_ctx = (FdContext*)events[i].data.ptr;
            
            // 4. 触发事件 = 恢复协程
            fd_ctx->triggerEvent(READ);  // 或 WRITE
        }
        
        // 5. yield() = 回到 Event Loop
        Fiber::GetThis()->yield();
    }
}
```

### 2.4.5 Reactor 在面试中的阐述

**面试问题：请解释 Reactor 模式**

**回答模板：**

> Reactor 是一种事件驱动的并发模式，将 IO 事件的检测和处理分离。
>
> **1. 核心组件：**
> - Handle：事件循环，持续等待事件
> - Event Demultiplexer：事件分离器（如 epoll）
> - Event Handlers：事件处理器
>
> **2. 工作流程：**
> 1. 注册事件处理器到 Handle
> 2. Handle 调用 epoll_wait 等待事件
> 3. 事件到来，Handle 调用对应的处理器
> 4. 处理器处理事件，完成后返回
> 5. 继续等待下一批事件
>
> **3. 变体模式：**
> - 单 Reactor 单线程：简单，但单核
> - 单 Reactor 多线程：Reactor 可能成为瓶颈
> - 主从 Reactor：Netty、Nginx 采用
>
> **4. 本项目的实现：**
> - 采用 Reactor + 协程的模式
> - 每个 IOManager 线程都是一个独立的 Reactor
> - epoll_wait 返回后，恢复对应的协程
> - 协程执行完后 yield，回到 Reactor 循环
> - 优势：既利用多核，又简化编程

## 2.5 RPC 基础知识深度讲解

### 2.5.1 什么是 RPC？

**RPC (Remote Procedure Call)** 远程过程调用，是一种**通过网络从远程计算机上请求服务**的技术，使得调用远程服务就像调用本地函数一样简单。

**本质问题：**

```
本地调用：
┌─────────────┐
│  Caller     │
│  result =   │
│  add(1, 2)  │
└──────┬──────┘
       │ 直接调用
       ▼
┌─────────────┐
│  Callee     │
│  return 3   │
└─────────────┘

远程调用：
┌─────────────┐                 ┌─────────────┐
│  Client     │                 │  Server     │
│             │    网络传输      │             │
│  stub.add() │ ──────────────► │  add(1, 2)  │
│             │  请求: {1, 2}    │             │
│  wait...    │◄──────────────── │  return 3   │
│             │  响应: {3}      │             │
│  result = 3 │                 │             │
└─────────────┘                 └─────────────┘

需要解决的问题：
1. 如何表示数据？ → 序列化
2. 如何传输数据？ → 网络协议
3. 如何找到服务器？ → 服务发现
4. 如何处理错误？ → 错误处理
5. 如何保证性能？ → 连接池、负载均衡
```

### 2.5.3 RPC 协议设计

**本项目的 RPC 协议格式：**

```
┌─────────────────────────────────────────────────────────────┐
│                    RPC 协议格式                              │
├─────────────────────────────────────────────────────────────┤
│  ┌────────────┐  ┌────────────┐  ┌────────────────────┐   │
│  │Total Length │  │Header Len  │  │    Header Data     │   │
│  │  (4 bytes)  │  │  (4 bytes) │  │  (Protobuf 格式)   │   │
│  └────────────┘  └────────────┘  └────────────────────┘   │
│                                                  │          │
│                                                  ▼          │
│                                     ┌────────────────────┐  │
│                                     │    Body Data       │  │
│                                     │  (Protobuf 格式)   │  │
│                                     └────────────────────┘  │
└─────────────────────────────────────────────────────────────┘

字段说明：
- Total Length: 整个消息的长度（不包含 Total Length 字段本身）
- Header Length: Header Data 的长度
- Header Data: 包含 service_name、method_name、args_size
- Body Data: 请求参数或响应数据

示例：
Total Length: 100
Header Length: 20
Header Data: {
  service_name: "UserService"
  method_name: "Login"
  args_size: 80
}
Body Data: {
  name: "zhangsan"
  pwd: "123456"
}
```

**为什么这样设计？**

```
1. 长度前缀解决粘包问题
   
   粘包问题：
   TCP 是流式协议，两次发送的数据可能粘在一起
   
   没有长度前缀：
   发送1: "Hello"
   发送2: "World"
   接收: "HelloWorld" → 无法区分两次发送
   
   有长度前缀：
   发送1: "5:Hello"
   发送2: "5:World"
   接收: 先读 5 字节 → "Hello"，再读 5 字节 → "World"

2. Header 和 Body 分离
   
   优点：
   - 可以先读取 Header，决定如何处理 Body
   - 可以流式读取大 Body
   - Header 和 Body 可以使用不同的序列化方式

3. 使用 Protobuf 序列化
   
   优点：
   - 高效的二进制格式
   - 跨语言支持
   - 自动生成代码
```

### 2.5.4 粘包/拆包问题

**问题描述：**

```
TCP 是流式协议，没有消息边界：

场景 1：粘包
发送端发送两个包：
  包1: "Hello"
  包2: "World"
接收端可能收到：
  "HelloWorld" → 两个包粘在一起

场景 2：拆包
发送端发送一个包：
  包1: "Hello World (100 字节)"
接收端可能分两次收到：
  第一次: "Hello World (50 字节)"
  第二次: "(50 字节)"

场景 3：粘包 + 拆包
发送端发送：
  包1: "Hello"
  包2: "Hello World (100 字节)"
接收端可能收到：
  第一次: "HelloHello World (50 字节)"
  第二次: "(50 字节)"
```

**解决方案：**

```cpp
// 方案 1：长度前缀（本项目采用）
struct RpcHeader {
    uint32_t total_length;  // 整个消息的长度
    uint32_t header_length; // header 的长度
    // header data
    // body data
};

// 发送
std::string send_data;
send_data.append((char*)&total_length, 4);
send_data.append((char*)&header_length, 4);
send_data.append(header_data);
send_data.append(body_data);
send(fd, send_data.c_str(), send_data.size(), 0);

// 接收
void recv_exact(int fd, void* buf, size_t size) {
    size_t received = 0;
    char* ptr = (char*)buf;
    while (received < size) {
        ssize_t n = recv(fd, ptr + received, size - received, 0);
        if (n > 0) {
            received += n;
        } else if (n == 0) {
            throw ConnectionClosed();
        } else {
            if (errno == EINTR) continue;
            throw RecvError();
        }
    }
}

// 读取 Total Length
uint32_t total_length;
recv_exact(fd, &total_length, 4);
total_length = ntohl(total_length);

// 读取 Header Length
uint32_t header_length;
recv_exact(fd, &header_length, 4);
header_length = ntohl(header_length);

// 读取 Header Data
std::string header_data(header_length, '\0');
recv_exact(fd, &header_data[0], header_length);

// 读取 Body Data
uint32_t body_length = total_length - 4 - header_length;
std::string body_data(body_length, '\0');
recv_exact(fd, &body_data[0], body_length);

// 方案 2：分隔符（如 HTTP 的 \r\n\r\n）
// 方案 3：固定长度
// 方案 4：消息头包含长度字段（如 HTTP 的 Content-Length）
```

### 2.5.5 服务注册与发现

**为什么需要服务注册与发现？**

```
问题：客户端如何知道服务器的地址？

方案 1：硬编码
string server_ip = "192.168.1.10:8080";
问题：
- 服务器宕机，客户端无法感知
- 无法动态扩容
- 无法负载均衡

方案 2：配置文件
[server]
ip = "192.168.1.10:8080"
问题：
- 修改配置需要重启客户端
- 无法自动感知服务器变化

方案 3：服务注册与发现（采用）
- 服务启动时注册到注册中心
- 客户端从注册中心获取地址
- 服务宕机时自动下线
- 支持动态扩容
```

**ZooKeeper 实现原理：**

```
ZooKeeper 节点结构：
/UserServiceRpc                    (持久节点)
  /Login                           (持久节点)
    /provider-0000000001            (临时顺序节点)
      数据: 192.168.1.10:8000
    /provider-0000000002            (临时顺序节点)
      数据: 192.168.1.11:8000

节点类型：
- 持久节点 (PERSISTENT)：显式删除才消失
- 临时节点 (EPHEMERAL)：客户端断开时自动删除
- 顺序节点 (SEQUENTIAL)：自动追加序号

服务注册流程：
1. Provider 启动
2. 连接 ZooKeeper
3. 创建 /UserServiceRpc (持久节点)
4. 创建 /UserServiceRpc/Login (持久节点)
5. 创建 /UserServiceRpc/Login/provider-* (临时节点)
6. 写入自己的 IP:Port
7. 如果宕机，临时节点自动删除

服务发现流程：
1. Client 调用 RPC
2. 连接 ZooKeeper
3. 查询 /UserServiceRpc/Login 的子节点
4. 获取所有 provider 的数据（IP:Port 列表）
5. 负载均衡选择一个
6. 发起调用
```

### 2.5.6 负载均衡策略

**常见策略对比：**

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **随机** | 简单 | 可能不均匀 | 节点性能相近 |
| **轮询** | 均匀 | 无法处理性能差异 | 节点性能相近 |
| **加权轮询** | 考虑性能 | 需要配置权重 | 节点性能不同 |
| **最少连接** | 动态平衡 | 需要维护连接数 | 长连接场景 |
| **一致性哈希** | 稳定 | 复杂 | 缓存场景 |

**本项目的实现：**

```cpp
// 随机负载均衡
std::string KrpcChannel::QueryServiceHost(...) {
    // 1. 检查本地缓存
    if (g_service_cache.count(method_path)) {
        sub_nodes = g_service_cache[method_path];
    }
    
    // 2. 缓存未命中，从 ZK 获取
    if (sub_nodes.empty()) {
        sub_nodes = zkclient->GetChildren(method_path);
        g_service_cache[method_path] = sub_nodes;
    }
    
    // 3. 随机选择
    static std::mt19937 rng(std::time(nullptr));
    std::uniform_int_distribution<int> dist(0, sub_nodes.size() - 1);
    int rand_idx = dist(rng);
    
    // 4. 获取节点数据
    std::string selected_node = sub_nodes[rand_idx];
    std::string host_data = zkclient->GetData(selected_node);
    
    return host_data;  // "192.168.1.10:8000"
}
```

### 2.5.7 连接池详解

**为什么需要连接池？**

```
问题：每次 RPC 调用都创建新连接

无连接池：
调用 1: connect() → send() → recv() → close()  ← 三次握手 + 四次挥手
调用 2: connect() → send() → recv() → close()  ← 三次握手 + 四次挥手
调用 3: connect() → send() → recv() → close()  ← 三次握手 + 四次挥手

问题：
- 每次 TCP 握手消耗时间 (~1ms)
- 每个 socket 占用文件描述符
- 无法复用连接

有连接池：
初始化: 创建 10 个连接放入池中
调用 1: GetConnection() → send() → recv() → ReleaseConnection()
调用 2: GetConnection() → send() → recv() → ReleaseConnection()
调用 3: GetConnection() → send() → recv() → ReleaseConnection()

优势：
- 复用连接，减少握手开销
- 减少文件描述符占用
- 提高性能
```

**连接池实现：**

```cpp
class ConnectionPool {
private:
    std::mutex _mutex;
    std::unordered_map<std::string, std::queue<int>> _conns;

public:
    int GetConnection(const std::string& ip, uint16_t port) {
        std::string key = ip + ":" + std::to_string(port);
        
        std::unique_lock<std::mutex> lock(_mutex);
        
        // 1. 尝试从池中获取
        if (_conns.find(key) != _conns.end() && !_conns[key].empty()) {
            int fd = _conns[key].front();
            _conns[key].pop();
            lock.unlock();
            
            // 检查连接是否有效
            if (is_connection_alive(fd)) {
                return fd;
            } else {
                close(fd);
            }
        }
        lock.unlock();
        
        // 2. 池中没有，创建新连接
        int clientfd = socket(AF_INET, SOCK_STREAM, 0);
        struct sockaddr_in server_addr;
        server_addr.sin_family = AF_INET;
        server_addr.sin_port = htons(port);
        server_addr.sin_addr.s_addr = inet_addr(ip.c_str());
        
        connect(clientfd, (struct sockaddr *)&server_addr, sizeof(server_addr));
        return clientfd;
    }
    
    void ReleaseConnection(const std::string& ip, uint16_t port, int fd) {
        std::string key = ip + ":" + std::to_string(port);
        
        std::lock_guard<std::mutex> lock(_mutex);
        _conns[key].push(fd);
    }
    
private:
    bool is_connection_alive(int fd) {
        // 发送心跳包检测
        // 或设置 SO_KEEPALIVE
        return true;
    }
};
```

**连接池需要注意的问题：**

```
1. 连接泄漏
   - 获取连接后忘记归还
   - 解决：使用 RAII，析构时自动归还

2. 连接失效
   - 长时间未使用，服务端关闭连接
   - 解决：定期检测，或在使用时检测

3. 连接池大小
   - 太小：等待连接，性能下降
   - 太大：占用资源
   - 解决：动态调整，根据负载自动扩缩容

4. 线程安全
   - 多线程同时访问连接池
   - 解决：加锁保护
```

### 3.1.2 为什么使用协程？

**对比三种并发模型：**

```cpp
// 模型 1：阻塞 IO + 多线程

    char buf[4096];
    int n = read(fd, buf, sizeof(buf));  // 阻塞当前线程
    process(buf, n);

// 问题：10,000 个连接需要 10,000 个线程

        process(buf, n);
        // 继续等待下一次可读
        epoll_ctl(..., EPOLL_CTL_MOD, fd, ...);
    }
}
// 问题：回调地狱，代码难以理解

// 模型 3：协程 (本项目)
void handle_client(int fd) {
    char buf[4096];
    int n = read(fd, buf, sizeof(buf));  // 看起来同步，实际异步
    if (n > 0) {
        process(buf, n);
    }
}

## 3.3 核心数据结构深度解析

### 3.3.1 Fiber 类详解

```cpp
// [fiber.h:13-49] Fiber 类定义
class Fiber: public std::enable_shared_from_this<Fiber>
{
public:
    enum State {
        READY,   // 就绪态：等待被调度
        RUNNING, // 运行态：正在执行
        TERM     // 终止态：执行完毕
    };

private:
    uint64_t m_id = 0;                    // 协程唯一 ID
    uint32_t m_stacksize = 0;             // 栈大小（字节）
    State m_state = READY;                // 当前状态
    ucontext_t m_ctx;                     // 协程上下文
    void* m_stack = nullptr;              // 协程栈指针
    std::function<void()> m_cb;           // 协程回调函数
    bool m_runInScheduler;                // 是否在调度器中运行
    std::mutex m_mutex;                   // 协程锁
};
```

**为什么这样设计？**

| 字段 | 设计理由 | 数据流转 |
|------|---------|---------|
| `m_id` | 唯一标识，便于调试和日志 | 创建时分配，生命周期内不变 |
| `m_stacksize` | 可配置栈大小，适应不同场景 | 构造时设置，128KB 默认值 |
| `m_state` | 状态机，防止非法操作 | READY→RUNNING→TERM |
| `m_ctx` | 保存/恢复执行上下文 | swapcontext 时读写 |
| `m_stack` | 协程栈空间，独立于线程栈 | malloc 分配，free 释放 |
| `m_cb` | 协程要执行的逻辑 | MainFunc 中调用 |
| `m_runInScheduler` | 区分调度协程/主协程 | 决定 resume/yield 的目标 |

### 3.3.2 线程局部存储详解

```cpp
// [fiber.cpp:5-9] TLS 变量
static thread_local Fiber* t_fiber = nullptr;
// → 当前线程正在运行的协程

static thread_local std::shared_ptr<Fiber> t_thread_fiber = nullptr;
// → 当前线程的主协程（线程启动时自动创建）

static thread_local Fiber* t_scheduler_fiber = nullptr;
// → 当前线程的调度协程（仅调度器线程中有效）
```

**为什么使用 thread_local？**

```
问题场景：
Thread 1 和 Thread 2 同时调用 Fiber::GetThis()

不使用 thread_local：
static Fiber* t_fiber = nullptr;  // 全局共享
Thread 1: t_fiber = fiber1;
Thread 2: t_fiber = fiber2;  // 覆盖了 Thread 1 的值！
结果：Thread 1 的协程状态错乱

使用 thread_local：
每个线程独立的 t_fiber
Thread 1: t_fiber = fiber1;  // Thread 1 的 t_fiber
Thread 2: t_fiber = fiber2;  // Thread 2 的 t_fiber
结果：互不干扰

内存布局：
Thread 1 的 TLS 区域    Thread 2 的 TLS 区域
┌─────────────────┐    ┌─────────────────┐
│ t_fiber → Fiber1 │    │ t_fiber → Fiber2 │
│ t_thread_fiber   │    │ t_thread_fiber   │
│ t_scheduler_fiber│    │ t_scheduler_fiber│
└─────────────────┘    └─────────────────┘
```

// [fiber.cpp:151-165] reset 实现
void Fiber::reset(std::function<void()> cb)
{
    // 必须是终止态才能重置
    assert(m_stack != nullptr && m_state == TERM);
    
    // 重新设置回调
    m_cb = cb;
    m_state = READY;
    
    // 重新初始化上下文（复用栈空间）
    getcontext(&m_ctx);
    m_ctx.uc_link = nullptr;
    m_ctx.uc_stack.ss_sp = m_stack;     // 复用原有栈
    m_ctx.uc_stack.ss_size = m_stacksize;
    makecontext(&m_ctx, &Fiber::MainFunc, 0);
}
```

**复用的优势：**

```
不复用：
创建协程 → 执行 → 销毁 → malloc(栈) → 执行 → free(栈) → ...
性能：每次 malloc/free 开销

复用：
创建协程 → 执行 → reset() → 执行 → reset() → 执行 → ...
性能：只有一次 malloc，后续零开销

内存对比：
不复用：10000 次创建 = 10000 * 128KB = 1.28GB 分配
复用：10000 次创建 = 10000 次栈复用 = 0 额外分配
```

}

### 3.5.3 MainFunc 详解

```cpp
// [fiber.cpp:137-150] MainFunc 实现
void Fiber::MainFunc()
{
    // 1. 获取当前协程的 shared_ptr，增加引用计数
    std::shared_ptr<Fiber> curr = GetThis();
    
    // 2. 执行用户函数
    curr->m_cb();
    
    // 3. 清理回调函数
    curr->m_cb = nullptr;
    curr->m_state = TERM;
    
    // 4. 引用计数管理的精妙之处
    auto raw_ptr = curr.get();
    curr.reset();  // 减少 shared_ptr 引用计数
    
    // 5. 让出执行权
    raw_ptr->yield();
}
```

**引用计数的精妙设计：**

```
引用计数变化：

┌─────────────────────────────────────────────────────────────┐
│  初始状态：调度器持有 shared_ptr<Fiber>                     │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ std::shared_ptr<Fiber> task = ...;                     ││
│  │ 引用计数 = 1 (调度器持有)                               ││
│  └─────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  resume() 调用 GetThis()                                   │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ std::shared_ptr<Fiber> curr = GetThis();               ││
│  │ return shared_from_this();                             ││
│  │ 引用计数 = 2 (调度器 + curr)                           ││
│  └─────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  MainFunc 中执行完毕                                        │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ curr->m_cb = nullptr;                                  ││
│  │ curr->m_state = TERM;                                  ││
│  │ auto raw_ptr = curr.get();                             ││
│  │ curr.reset();  // 引用计数 = 1 (调度器仍持有)          ││
│  │ raw_ptr->yield();  // 通过原始指针调用，对象仍有效     ││
│  └─────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  回到调度器                                                 │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ task->getState() == TERM                              ││
│  │ task->reset(new_cb);  // 可以复用                      ││
│  └─────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────┘

为什么这样设计？
1. shared_from_this() 增加引用计数，确保对象在执行期间不被销毁
2. curr.reset() 减少引用计数，但对象仍有效（调度器持有）
3. raw_ptr->yield() 安全调用，因为对象不会立即销毁
4. 回到调度器后，可以安全地复用对象
```

### 4.2.2 任务调度流程

```cpp
// [scheduler.cpp:66-147] run 实现核心循环
void Scheduler::run()
{
    // 每个工作线程都执行这个循环
    while(true) {
        ScheduleTask task;
        bool tick_me = false;
        
        // 1. 从任务队列取任务
        {
            std::lock_guard<std::mutex> lock(m_mutex);
            auto it = m_tasks.begin();
            
            while(it != m_tasks.end()) {
                // 跳过指定给其他线程的任务
                if(it->thread != -1 && it->thread != thread_id) {
                    it++;
                    tick_me = true;
                    continue;
                }
                
                task = *it;
                m_tasks.erase(it);
                m_activeThreadCount++;
                break;
            }
        }
        
        // 2. 执行任务
        if(task.fiber) {
            task.fiber->resume();
            m_activeThreadCount--;
        } else if(task.cb) {
            std::shared_ptr<Fiber> task_fiber = std::make_shared<Fiber>(task.cb);
            task_fiber->resume();
            m_activeThreadCount--;
        } else {
            // 无任务，执行 idle 协程
            m_idleThreadCount++;
            idle_fiber->resume();
            m_idleThreadCount--;
        }
    }
}
```

## 4.3 M:N 调度模型深度解析

### 4.3.1 为什么是 M:N？

```
M:N 的含义：
M = 协程数量（用户态并发单元）
N = 线程数量（内核态执行单元）

为什么不是 1:1？
- 每个协程一个线程 → 开销大，无法创建大量协程

为什么不是 M:1？
- 所有协程在一个线程 → 无法利用多核

为什么是 M:N？
- 协程数量 >> 线程数量
- 充分利用多核
- 高并发能力
```

### 4.3.2 调度算法详解

```
调度算法：

while(true) {
    // 1. 优先级调度
    if (有高优先级任务) {
        取高优先级任务;
    } else {
        取普通任务;
    }
    
    // 2. 任务窃取
    if (当前线程队列为空 && 其他线程队列不空) {
        从其他线程窃取任务;
    }
    
    // 3. 公平调度
    if (任务运行时间过长) {
        挂起当前任务，调度其他任务;
    }
    
    // 4. 执行任务
    resume(任务);
}
```

### 4.3.3 负载均衡

```
负载均衡策略：

┌─────────────────────────────────────────────────────────────┐
│  线程 1                    线程 2                    线程 3    │
│  ┌──────────┐             ┌──────────┐             ┌───────┐  │
│  │ Task1    │             │ Task4    │             │Task7 │  │
│  │ Task2    │             │ Task5    │             │Task8 │  │
│  │ Task3    │             │ Task6    │             │Task9 │  │
│  └──────────┘             └──────────┘             └───────┘  │
│      3 任务                  3 任务                  3 任务   │
└─────────────────────────────────────────────────────────────┘

不均衡情况：

{

        Fiber::GetThis()->yield();
    }

### 4.5.1 当前实现的优缺点

// [ioscheduler.cpp:90-151] addEvent 实现
int IOManager::addEvent(int fd, Event event, std::function<void()> cb)
{
    // 1. 获取或创建 FdContext
    FdContext* fd_ctx = m_fdContexts[fd];
    if(!fd_ctx) {
        fd_ctx = new FdContext();
        m_fdContexts[fd] = fd_ctx;
    }
    
    // 2. 检查事件是否已存在
    std::unique_lock<std::mutex> lock(fd_ctx->mutex);
    if(fd_ctx->events & event) {
        return -1;
    }
    
    // 3. 构建 epoll_event
    epoll_event ev;
    ev.events = fd_ctx->events | event;
    ev.data.ptr = fd_ctx;
    
    // 4. 调用 epoll_ctl
    int op = fd_ctx->events ? EPOLL_CTL_MOD : EPOLL_CTL_ADD;
    epoll_ctl(m_epfd, op, fd, &ev);
    
    // 5. 更新 FdContext
    fd_ctx->events = (Event)(fd_ctx->events | event);
    
    // 6. 设置事件上下文
    FdContext::EventContext& event_ctx = fd_ctx->getEventContext(event);
    event_ctx.scheduler = Scheduler::GetThis();
    
    if(cb) {
        event_ctx.cb = cb;
    } else {
        event_ctx.fiber = Fiber::GetThis();
    }
    
    return 0;
}

// [ioscheduler.cpp:27-46] triggerEvent 实现
void IOManager::FdContext::triggerEvent(Event event)
{
    // 1. 检查事件是否注册
    assert(events & event);
    
    // 2. 清除事件标记
    events = (Event)(events & ~event);
    
    // 3. 获取事件上下文
    EventContext& temp = getEventContext(event);
    
    // 4. 调度到调度器
    if(temp.cb) {
        // 回调函数：直接调度
        temp.scheduler->scheduleLock(temp.cb);
    } else {
        // 协程：恢复执行
        temp.scheduler->scheduleLock(temp.fiber);
    }
    
    // 5. 重置事件上下文
    resetEventContext(temp);
}
```

{

extern "C" unsigned int sleep(unsigned int seconds)
{
    if(!t_hook_enable) {
        return sleep_f(seconds);  // 直接调用原始 sleep
    }
    
    std::shared_ptr<Fiber> fiber = sylar::Fiber::GetThis();
    sylar::IOManager* iom = sylar::IOManager::GetThis();
    
    // 添加定时器，到期后恢复协程
    iom->addTimer(seconds * 1000, [fiber, iom](){

    });
    
    // 挂起协程
    fiber->yield();
    
    return 0;
}

int connect_with_timeout(int fd, const struct sockaddr* addr, 
                         socklen_t addrlen, uint64_t timeout_ms) 
{
    // 1. 尝试连接
    int n = connect_f(fd, addr, addrlen);
    
    if(n == 0) {
        return 0;  // 立即成功
    } else if(n != -1 || errno != EINPROGRESS) {

    // 6. 被唤醒后检查连接结果
    int error = 0;
    socklen_t len = sizeof(int);
    getsockopt(fd, SOL_SOCKET, SO_ERROR, &error, &len);
    
    if(!error) {
        return 0;  // 连接成功
    } else {
        errno = error;
        return -1;  // 连接失败
    }
}

    m_listenFd = socket(AF_INET, SOCK_STREAM, 0);
    
    // 2. 设置 SO_REUSEADDR
    int val = 1;
    setsockopt(m_listenFd, SOL_SOCKET, SO_REUSEADDR, &val, sizeof(val));
    
    // 3. 设置为阻塞模式（配合 Hook）
    int listen_flags = fcntl(m_listenFd, F_GETFL, 0);
    fcntl(m_listenFd, F_SETFL, listen_flags & ~O_NONBLOCK);
    
    // 4. 绑定地址
    sockaddr_in addr{};
    addr.sin_family = AF_INET;
    addr.sin_port = htons(port);
    addr.sin_addr.s_addr = INADDR_ANY;
    
    ::bind(m_listenFd, (sockaddr*)&addr, sizeof(addr));
    
    return true;
}

// [http_server.cpp:76-115] handleAccept 实现
void HttpServer::handleAccept()
{
    while(!m_stop) {
        // 1. accept 新连接
        int clientFd = accept(m_listenFd, nullptr, nullptr);
        
        if(clientFd > 0) {
            // 2. 设置为阻塞模式
            int client_flags = fcntl(clientFd, F_GETFL, 0);
            fcntl(clientFd, F_SETFL, client_flags & ~O_NONBLOCK);
            
            // 3. 调度处理协程
            m_worker->scheduleLock([self = shared_from_this(), clientFd](){
                self->handleClient(clientFd);
            });
        } else {
            // 4. 没有更多连接了
            if(errno != EAGAIN && errno != EWOULDBLOCK) {
                SYLAR_LOG_ERROR("Accept error");
            }
            break;
        }
    }
    
    // 5. 重新注册事件（Re-arm）
    if(!m_stop) {
        m_worker->addEvent(m_listenFd, IOManager::READ, [this](){
            this->handleAccept();
        });
    }
}

// [http_server.cpp:117-189] handleClient 实现
void HttpServer::handleClient(int client_fd)
{
    std::vector<char> buf(4096);
    
    while(true) {
        // 1. 读取请求（Hook 后自动挂起）
        int n = read(clientFd, buf.data(), buf.size());
        
        if(n > 0) {
            // 2. 解析请求
            HttpRequest::ptr req(new HttpRequest());
            req->parse(std::string(buf.data(), n));
            
            // 3. 构建响应
            HttpResponse::ptr resp(new HttpResponse());
            resp->status = 200;
            resp->headers["Server"] = "Sylar/1.0";
            
            // 4. 路由分发
            if(req->path == "/rpc/login") {
                static sylar::http::RpcServlet rpc_servlet;
                rpc_servlet.handle(req, resp, nullptr);
            }
            
            // 5. 发送响应
            std::string response_str = resp->toString();
            send(client_fd, response_str.c_str(), response_str.size(), 0);
        }
        else if(n == 0) {
            break;  // 连接关闭
        }
        else {
            break;
        }
    }
    
    close(client_fd);
}

## 7.3 对比分析与改进方向

### 7.3.1 当前实现的优缺点

**优点：**

1. **简洁**：代码逻辑清晰，易理解
2. **协程化**：每个连接一个协程，异步同步写法
3. **Reactor 模式**：事件驱动，高性能

**缺点：**

1. **单次读取**：假设请求小于 4KB
2. **无 keep-alive**：每次请求后关闭连接
3. **无缓存**：静态资源每次都处理
4. **无限流**：可能被大流量打垮

### 7.3.2 改进方向

**改进 1：循环读取**

```cpp
// 当前：一次读取

// 改进：循环读取直到完整请求
std::string request;
while(true) {
    char buf[4096];
    int n = read(clientFd, buf, sizeof(buf));
    if(n > 0) {
        request.append(buf, n);
        // 检查是否收到完整请求（\r\n\r\n）
        if(request.find("\r\n\r\n") != std::string::npos) {
            break;
        }
    } else if(n == 0) {
        break;
    } else {
        break;
    }
}
```

class ConnectionPool {
    std::map<int, HttpConnection> m_connections;
    
    void handleClient(int fd) {
        auto& conn = m_connections[fd];
        while(true) {
            // 读取请求
            HttpRequest::ptr req = conn.readRequest();
            if(!req) break;
            
            // 处理请求
            HttpResponse::ptr resp = processRequest(req);
            conn.sendResponse(resp);

            }
        }
        close(fd);
        m_connections.erase(fd);
    }
};

└─────────────────────────────────────────────────────────────┘

                          ▼
┌─────────────────────────────────────────────────────────────┐
│  epoll_wait 返回                                               │
│  int rt = epoll_wait(m_epfd, events, MAX_EVENTS, timeout);    │
│  返回 1 个事件                                                 │
│  events[0].data.fd = client_fd                                │
│  events[0].events = EPOLLIN                                  │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐

│  for(int i = 0; i < rt; ++i) {                              │
│      FdContext* fd_ctx = (FdContext*)events[i].data.ptr;       │
│      fd_ctx->triggerEvent(READ);                              │
│  }                                                          │
└─────────────────────────────────────────────────────────────┘
                          │
                          │ triggerEvent
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  void FdContext::triggerEvent(Event event) {                   │
│      // 1. 检查事件                                           │
│      assert(events & event);                                 │
│      events = (Event)(events & ~event);                        │
│                                                             │
│      // 2. 获取事件上下文                                   │
│      EventContext& temp = getEventContext(event);               │
│                                                             │
│      // 3. 调度到调度器                                       │
│      if(temp.cb) {                                           │
│          temp.scheduler->scheduleLock(temp.cb);                 │
│      } else {                                                │
│          temp.scheduler->scheduleLock(temp.fiber);               │
│      }                                                       │
│                                                             │
│      // 4. 重置事件上下文                                   │
│      resetEventContext(temp);                                │
│  }                                                          │
└─────────────────────────────────────────────────────────────┘
                          │
                          │ scheduleLock
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  任务被加入调度队列                                           │
│  m_tasks.push_back(ScheduleTask(handleClient_fiber));        │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  调度器取出任务并执行                                         │
│  handleClient_fiber->resume();                             │
│                                                             │
│  resume() 内部：                                            │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ swapcontext(&scheduler_ctx, &fiber_ctx);                ││
│  │  - 保存调度协程的寄存器到 scheduler_ctx              ││
│  │  - 恢复 handleClient 协程的寄存器从 fiber_ctx              ││
│  └─────────────────────────────────────────────────────────┘│
│                                                             │
│  handleClient 协程从 yield() 后继续执行                        │
└─────────────────────────────────────────────────────────────┘
                          │
                          │ 回到 retry 标签
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  retry:                                                     │
│  ssize_t n = read_f(fd, buf, size);                         │
│  // 这次成功！                                                │
│  // 返回实际读取的字节数                                       │
└─────────────────────────────────────────────────────────────┘
```

## 9.1 协程相关面试题

### Q1: 什么是协程？协程和线程有什么区别？

**标准答案：**

> 协程是用户态的轻量级线程，由程序自己控制调度，无需内核介入。
>
> **核心区别：**
> | 特性 | 协程 | 线程 |
> |------|------|------|
> | 调度方式 | 用户态调度 | 内核态调度 |
> | 切换开销 | ~50ns | ~1-5μs |
> | 栈大小 | 2KB-128KB | 8MB |
> | 数量 | 单机 10万+ | 单机几百到几千 |
> | 编程模型 | 同步代码写异步逻辑 | 需要回调/状态机 |
>
> **本项目中的实现：**
> - 基于 Linux 的 ucontext_t 实现用户态上下文切换
> - 采用 M:N 调度模型，4 个线程运行 10000+ 协程
> - 通过 swapcontext() 保存/恢复 CPU 寄存器实现切换

### Q2: 协程切换的底层原理是什么？

**标准答案：**

> 协程切换通过保存和恢复执行上下文实现。
>
> **核心函数：**
> - `getcontext()`: 获取当前上下文
> - `makecontext()`: 初始化新上下文
> - `swapcontext(&ouctx, &inctx)`: 保存当前并切换到目标
>
> **切换流程：**
> 1. swapcontext 被调用时，将当前 CPU 寄存器保存到 ouctx
> 2. 从 inctx 恢复寄存器到 CPU
> 3. 跳转到 inctx 的 RIP 指针继续执行
>
> **寄存器保存内容：**
> - RIP（指令指针）
> - RSP（栈指针）
> - RBP（基址指针）
> - RBX, R12-R15（通用寄存器）
> - 信号掩码
>
> **本项目实现：**
> ```cpp
> // [fiber.cpp:84-107] resume 实现
> void Fiber::resume() {
>     SetThis(this);
>     swapcontext(&t_scheduler_fiber->m_ctx, &m_ctx);
> }
> ```

### Q3: 为什么要使用 thread_local 存储协程指针？

**标准答案：**

> 因为每个线程有独立的协程栈，thread_local 确保线程间互不干扰。
>
> **三个 TLS 变量的作用：**
> - `t_fiber`: 当前线程正在运行的协程
> - `t_thread_fiber`: 当前线程的主协程
> - `t_scheduler_fiber`: 当前线程的调度协程
>
> **为什么不使用全局变量？**
> - 全局变量会导致线程安全问题
> - 多线程同时访问需要加锁，影响性能
> - thread_local 无锁，性能更好
>
> **本项目实现：**
> ```cpp
> // [fiber.cpp:5-9]
> static thread_local Fiber* t_fiber = nullptr;
> static thread_local std::shared_ptr<Fiber> t_thread_fiber = nullptr;
> static thread_local Fiber* t_scheduler_fiber = nullptr;
> ```

### Q4: 协程如何复用？为什么要复用？

**标准答案：**

> 协程执行完后不销毁，而是通过 reset() 重新初始化，复用栈空间。
>
> **复用流程：**
> 1. 协程执行完毕，状态变为 TERM
> 2. 调度器检查状态，发现可以复用
> 3. 调用 reset() 设置新的回调函数
> 4. 状态变为 READY，可以再次 resume()
>
> **复用的优势：**
> - 减少 malloc/free 开销
> - 避免内存碎片
> - 提高性能
>
> **本项目实现：**
> ```cpp
> // [fiber.cpp:151-165] reset 实现
> void Fiber::reset(std::function<void()> cb) {
>     assert(m_state == TERM);  // 必须是终止态
>     m_cb = cb;
>     m_state = READY;
>     makecontext(&m_ctx, &Fiber::MainFunc, 0);  // 重新设置入口
> }
> ```

>

>     }

>

> - 事件驱动，而非轮询
> - 结合定时器和 IO 事件
> - 更高效的 CPU 利用

## 9.3 Epoll 相关面试题

### Q1: epoll LT vs ET 的区别？

**标准答案：**

> | 特性 | LT（水平触发） | ET（边缘触发） |
> |------|---------------|--------------|
> | 触发条件 | 缓冲区有数据就通知 | 状态变化时通知一次 |
> | 编程难度 | 简单 | 复杂 |
> | 性能 | 较低（频繁唤醒） | 较高（减少唤醒） |
> | 读取要求 | 可分次读取 | 必须一次读完 |
>
> **本项目选择 ET 模式的原因：**
> - 性能更好，减少系统调用
> - 配合 Hook 机制使用
> - 代码已正确处理 ET 模式（循环 read 直到 EAGAIN）

### Q2: 为什么边缘触发要一次读完？

**标准答案：**

> ET 模式只在状态变化时通知一次，如果不一次性读完：
> - 剩余数据不会再次触发通知
> - 导致数据丢失
>
> **正确的 ET 处理方式：**
> ```cpp
> while(true) {
>     int n = read(fd, buf, size);
>     if(n > 0) {
>         // 处理数据
>     } else if(n == 0) {
>         break;  // 连接关闭
>     } else {
>         if(errno == EAGAIN) {
>             break;  // 数据读完了
>         }
>         break;
>     }
> }
> ```

### Q3: epoll_wait 的 timeout 参数如何设置？

**标准答案：**

> timeout 应该设置为下一个定时器的超时时间。
>
> **本项目实现：**
> ```cpp
> // [ioscheduler.cpp:307-311]
> uint64_t nextTimer = getNextTimer();
> uint64_t timeOut = std::min(5000ull, nextTimer);
> int rt = epoll_wait(m_epfd, events.get(), MAX_EVENTS, timeOut);
> ```
>
> **为什么这样设计？**
> - 如果没有定时器，可以设置一个较大的超时（如 5 秒）
> - 如果有待触发的定时器，timeout 设为最近的定时器时间
> - 定时器到期时，epoll_wait 返回，处理定时器
> - 这样保证了定时器的精确性

## 9.4 Hook 机制面试题

### Q1: 如何实现函数 Hook？

**标准答案：**

> 使用 dlsym 和动态链接机制。
>
> **实现步骤：**
> 1. 定义原始函数指针：`read_fun read_f = nullptr;`
> 2. 在全局对象的构造函数中调用 dlsym：
>    ```cpp
>    read_f = (read_fun)dlsym(RTLD_NEXT, "read");
>    ```
> 3. 定义同名函数覆盖原始函数
>
> **dlsym(RTLD_NEXT) 的作用：**
> - 查找系统中下一个同名符号
> - 即 glibc 中的原始 read 函数
>
> **代码示例：**
> ```cpp
> extern "C" ssize_t read(int fd, void *buf, size_t count) {
>     return do_io(fd, read_f, "read", READ, SO_RCVTIMEO, buf, count);
> }
> ```

### Q2: Hook 后的 read 为什么是异步的？

**标准答案：**

> 因为 Hook 在 EAGAIN 时注册了 epoll 事件并挂起协程，而不是阻塞线程。
>
> **完整流程：**
> 1. read() 调用 read_f() 返回 EAGAIN
> 2. addEvent(READ) 注册读事件
> 3. yield() 挂起协程（线程继续运行其他协程）
> 4. 数据到达时，epoll_wait 返回
> 5. triggerEvent() 恢复协程
> 6. 重试 read()，这次成功
>
> **优势：**
> - 线程不阻塞，可以处理其他连接
> - 同步代码写法，异步执行效果

### Q3: connect 为什么要处理 EINPROGRESS？

**标准答案：**

> 非阻塞 connect() 立即返回 EINPROGRESS，表示正在连接中。
>
> **连接建立过程：**
