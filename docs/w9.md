## 4. 关键设计

### 4.1 Router + Servlet
新增 `Router` 与 `Servlet` 抽象，支持：
1. 精确路由：`method + normalized_uri`。
2. 前缀路由：`/path-*`（以 `*` 结尾）用于可控的模式匹配。
3. URI 规范化：
- 去掉 query/hash（例如 `/rpc/invoke?x=1` -> `/rpc/invoke`）。
- 收敛多余斜杠。
- 统一尾斜杠（`/rpc/invoke/` -> `/rpc/invoke`）。

稳定命中策略：
1. 先查精确路由（method 精确，再 `*` 方法兜底）。
2. 再查前缀路由（按前缀长度降序，保证最具体规则优先）。

### 4.2 鉴权拦截器（API Key/JWT）
新增 `AuthInterceptor`，采用 OR 语义：
1. `X-API-Key` 命中配置值可放行。
2. `Authorization: Bearer <jwt>` 通过校验可放行。
3. 其余情况返回 `401 Unauthorized`。

JWT 校验（当前工程版本）：
1. 校验 token 结构（三段式）。
2. Base64URL 解码 payload。
3. 校验 `iss/aud/exp`（若配置要求）。
4. 当前不做签名验签（用于工程联调与测试场景）。

### 4.3 鉴权审计
拦截器维护审计快照：
1. `unauthorized_total`
2. `api_key_rejected`
3. `jwt_rejected`
4. `jwt_expired`
5. `last_reject_reason`

这样可直接用于“未授权请求是否被拦截并可观测”的验收。

### 4.4 网关错误码映射
新增 `map_rpc_code_to_http_status(int rpc_code)`：
1. `0 -> 200`
2. `4xx -> 原样透传`
3. `503/504 -> 原样透传`
4. 其他 `5xx/未知错误 -> 502 Bad Gateway`

收益：
1. 上游客户端看到统一 HTTP 语义。
2. 网关与 RPC 内部错误码解耦。

### 4.5 网关与跨层依赖解耦
W9 后处理链路为：
1. `net/reactor_http_server` 仅负责 accept/read/write、HTTP 解析和响应写回。
2. 业务处理统一委托给 `gateway::HttpGateway`（Router/Servlet/Auth/错误映射）。
3. `gateway::HttpGateway` 内部仅依赖 `rpc::client::IRpcClient` 抽象接口调用上游。

这意味着 net 层不再内联业务逻辑，gateway 也不直接依赖 net 内部实现。

## 5. 与 Boost/Beast 的使用
W9 延续并复用 W8 的 Boost/Beast 能力：
1. `boost::beast::http` 继续用于 HTTP 报文封包与解析链路。
2. 新增 Router/Auth/RPC 映射逻辑叠加在 Beast 之上，不重复造 HTTP 协议轮子。
3. 连接读写路径从裸 `recv/send` 切换为 Beast/Asio `socket.receive/socket.send` 接口，
   在保留 epoll 主循环的前提下统一错误码语义与非阻塞处理分支。
4. `ReactorHttpServer::process_request` 改为值传递 + move 投影，减少 ingress task
   向 gateway 请求模型转换时的字符串与 header map 复制开销。
5. `HttpGateway` 热点路径优化：
- Header 查询增加“直接命中”快速路径，避免常见小写键名重复归一化分配。
- `/short`、`/slow`、`/keep-*` 统一改为预分配字符串拼接，去掉 `ostringstream`。
6. 每连接改为持有 Beast 增量 `request_parser + flat_buffer`，按 epoll 读事件持续推进解析状态，
   避免重复从字符串头部全量重解析。
7. 写路径改为 `write_offset` 推进发送，避免每次 `send` 后 `erase(0, n)` 的 O(n) 内存搬移。

## 9. 项目优点沉淀（W9）
1. Router + Servlet 把“路由匹配”和“业务处理”拆分，提升可维护性与可扩展性。
2. API Key/JWT 鉴权拦截统一前置，未授权请求可在入口层快速拒绝并审计。
3. RPC 错误到 HTTP 状态码映射统一了 4xx/5xx 语义，减少调用方歧义。
4. 网关仅依赖 `IRpcClient` 抽象接口，跨层耦合显著降低，替换实现更安全。

### 11.2 高频问答
1. 问：为什么网关要统一错误码语义。  
答：统一语义可让客户端按状态码稳定分支处理，降低跨服务认知成本。
2. 问：为什么鉴权要在 Router/Servlet 前置。  
答：前置拦截能减少无效业务计算，同时保证审计路径一致。
3. 问：为什么强调“网关只依赖 `IRpcClient`”。  
答：可替换 RPC 实现而不改网关业务逻辑，降低演进成本和回归风险。
