# W12 帮助文档（Raft 一致性替代 ZooKeeper）

## 1. 目标与范围
W12 目标是把元数据强依赖从外部 ZooKeeper 收敛到项目内自研 Raft 模块，承载两类核心场景：
1. 服务注册发现
2. 配置同步

本周实现聚焦四项能力：
1. Raft Leader 选举
2. 日志复制与状态机提交
3. 快照裁剪与落后节点快照安装
4. Joint Consensus 在线扩缩容

## 3. 设计说明

### 3.1 元数据状态机
Raft 日志承载以下命令：
1. `UpsertConfig` / `EraseConfig`
2. `RegisterService` / `UnregisterService`
3. `BeginJointConsensus` / `FinalizeJointConsensus`

状态机数据：
1. `configs`：配置键值表
2. `services`：服务名 -> 实例列表（`id/host/port`）

### 3.2 选举与复制
1. 选举超时与心跳间隔按 tick 驱动，支持 `wait_for_leader(max_ticks)` 收敛检查。
2. 写请求统一通过 Leader 追加日志并复制。
3. 提交判定：
- 常规配置：多数派
- Joint 配置：旧配置多数 + 新配置多数

### 3.3 快照机制
1. 当 `committed_index - snapshot_index >= snapshot_log_threshold` 时触发快照。
2. Leader 快照保存已提交状态机，并裁剪日志前缀。
3. 落后节点在复制时自动安装 Leader 快照后继续追日志。

### 3.4 Joint Consensus
接口：
1. `begin_joint_consensus(new_voters)`
2. `finalize_joint_consensus()`
3. `reconfigure_joint(new_voters, max_ticks)`（封装 begin + finalize）

语义：
1. Begin 提交后进入联合投票期
2. Finalize 提交后切换为新成员配置
3. 成员变更期间提交和选举都要求双多数，保证一致性

### 3.5 协程调度融合
`RaftOptions::scheduler` 可注入 `CoroutineScheduler`。
1. 注入并运行时：关键 Raft 操作通过调度器执行
2. 未注入或调度器未运行时：本地线程直执行业务
3. 观测字段 `scheduler_dispatches` 可用于验收“已融合协程链路”

### 3.6 与现有接口适配
1. `RaftServiceDiscovery`：适配 `IServiceDiscovery`
2. `RaftConfigCenterClient`：适配 `IConfigCenterClient`

用于平滑替换当前默认内存实现，减少外部依赖耦合。

## 4. 对外接口速查
`RaftCluster` 常用接口：
1. 生命周期：`bootstrap` / `tick` / `run_ticks` / `wait_for_leader`
2. 节点控制：`set_node_available`
3. 元数据写入：`upsert_config` / `erase_config` / `register_service` / `unregister_service`
4. 元数据读取：`config_value` / `config_snapshot` / `discover_service`
5. 成员变更：`begin_joint_consensus` / `finalize_joint_consensus` / `reconfigure_joint`
6. 验收观测：`metadata_consistent` / `status` / `leader`

## 5. 验收映射

### 5.1 验收项 1
节点变更与 Leader 切换在业务可接受时延内收敛，且业务流量无中断。

对应测试点（`w12_raft_consensus_test`）：
1. 3 节点集群在 `64` tick 内选出 Leader。
2. 隔离当前 Leader 后，`40` tick 内完成新 Leader 收敛。
3. Failover 期间持续读取服务发现，确保始终有返回节点。
4. 扩容到 4 节点、再缩容到 3 节点，变更后读写持续可用。

### 5.2 验收项 2
移除 ZooKeeper 外部强依赖后，元数据一致性与可用性满足集群要求。

对应测试点：
1. 状态机命令覆盖配置同步 + 服务注册发现。
2. `metadata_consistent()` 校验当前投票成员数据一致。
3. `RaftConfigCenterClient::fetch_latest()` 直接从 Raft 状态机产出快照。
4. `RaftServiceDiscovery::list_nodes()` 直接从 Raft 状态机产出服务节点。

## 7. 使用建议
1. 生产化前建议补齐：持久化 WAL、日志压缩、网络 RPC、故障注入回放。
2. 当前实现适合先作为“去外部依赖”的本地一致性骨架与联调替身。
3. 若需接入网关主链路，优先通过 `RaftServiceDiscovery` 与 `RaftConfigCenterClient` 注入，避免跨层直接依赖 Raft 内部结构。

## 9. 生产化补充内容

### 9.1 WAL / 快照落盘恢复
目标：节点进程重启后可恢复 `term/vote/log/snapshot/commit_index`，不丢已提交元数据。

建议新增模块：
1. `include/rpc/infra/raft/raft_storage.h`
2. `src/infra/raft/raft_storage.cpp`

建议接口：
1. `append_wal(const LogEntry&)`
2. `flush_wal()`
3. `save_snapshot(const Snapshot&)`
4. `load_latest_snapshot()`
5. `replay_wal_from(snapshot_index)`

实现要点：
1. WAL 采用顺序追加 + CRC 校验，按 segment 滚动。
2. 快照采用临时文件写入后原子重命名（防止半写文件）。
3. 启动恢复顺序：`snapshot -> wal replay -> rebuild state machine -> rejoin raft`。
4. 安装快照成功后裁剪旧 WAL segment，控制磁盘占用。

验收建议：
1. 重启前后 `committed_index` 单调不回退。
2. 重启后配置与服务注册数据与重启前一致。
3. 非正常退出（kill -9）后仍可恢复到最近可用状态。

### 9.2 真实节点间 Raft RPC
目标：把当前单进程仿真复制改为真实网络通信（AppendEntries / RequestVote / InstallSnapshot）。

建议新增模块：
1. `proto/raft.proto`
2. `include/rpc/infra/raft/raft_rpc.h`
3. `src/infra/raft/raft_rpc.cpp`
4. `include/rpc/infra/raft/raft_node.h`
5. `src/infra/raft/raft_node.cpp`

建议 RPC：
1. `RequestVote(term, candidate_id, last_log_index, last_log_term)`
2. `AppendEntries(term, leader_id, prev_log_index, prev_log_term, entries, leader_commit)`
3. `InstallSnapshot(term, leader_id, last_included_index, last_included_term, chunk...)`

实现要点：
1. 每个节点独立进程/线程实例，节点地址来自成员配置。
2. 发送端支持超时、重试、幂等请求标识；接收端保证 term 与日志一致性检查。
3. 网络异常时使用退避重试，避免对业务流量造成放大效应。

验收建议：
1. 3~5 节点真实网络下可稳定选主与提交。
2. 断连恢复后 follower 自动补齐日志或快照。
3. 成员变更期间双多数判定持续正确，无脏提交。

### 9.3 故障注入回归
目标：把一致性与可用性从“功能通过”提升为“故障场景可重复验证”。

建议新增测试：
1. `tests/integration/raft_partition_recovery_test.cpp`
2. `tests/integration/raft_restart_recovery_test.cpp`
3. `tests/integration/raft_joint_consensus_churn_test.cpp`

故障矩阵建议：
1. 网络分区：`leader minority`、`follower minority`、`split brain 窗口`。
2. 节点抖动：随机上下线、短周期重启、慢磁盘写入。
3. 变更叠加：Joint Consensus 过程中再触发 leader 切换。
4. 压力扰动：高频配置写入 + 服务注册变更并发。

核心观测指标：
1. `leader_election_converge_ms`
2. `log_replication_lag`
3. `snapshot_install_count`
4. `joint_consensus_duration_ms`
5. `metadata_consistency_check_pass_ratio`

回归门禁建议：
1. 关键故障用例连续通过（例如 100 轮无脏提交）。
2. Leader 切换与成员变更收敛时延稳定在预算阈值内。
3. 业务读路径（服务发现/配置读取）在故障窗口内不出现全量中断。
