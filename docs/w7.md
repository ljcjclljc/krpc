## 4. 关键设计与实现

### 4.1 超时分层（gateway -> rpc -> net）
1. Gateway 入口 `invoke_with_deadline(...)` 创建 `RequestContext`，并通过 `ScopedRequestContext` 绑定到当前线程上下文。
2. RPC 层将请求超时配置传入 net 层（`downstream_timeout_ms`）。
3. Net 层 `derive_effective_timeout(...)` 读取上游 `deadline`，计算：
- `effective_timeout_ms = min(上游剩余预算, 下游timeout配置)`；
- 若下游配置为 `0`，则仅使用上游剩余预算；
- 若上游预算已耗尽，则直接返回 `deadline_exceeded`（504）。
4. `invoke_stub(...)` 将本次生效预算写入 `NetCallResponse.effective_timeout_ms`，并暴露 `last_effective_timeout_ms()` 供测试校验跨层一致性。

### 4.2 重试预算（RetryBudget）
1. `RetryBudget` 使用滑动时间窗口维护请求时间戳与重试时间戳。
2. 默认参数：
- `window = 1000ms`
- `retry_ratio = 0.2`
- `min_retry_tokens = 1`
3. 每次 RPC 首发前调用 `record_request()` 累积请求基数。
4. 每次进入重试（`attempt > 1`）前先检查：
- 是否超过 `max_retries`；
- 是否能 `try_acquire_retry_token()`。
5. 预算不足时立即返回 `429 retry_budget_exhausted`，从而阻断无限重试链路。

### 4.3 观测信息
当前版本可用于验收的关键观测字段：
1. `RpcResponse.attempts`：本次调用总尝试次数。
2. `RpcResponse.effective_timeout_ms`：跨层预算生效值。
3. `RpcResponse.code/message`：例如 `429 retry_budget_exhausted`、`503 transient_upstream_error`、`504 deadline_exceeded`。
4. `rpc::net::last_effective_timeout_ms()`：用于验证 RPC 与 Net 层预算一致。

## 9. 复习提纲（建议背诵）

### 9.1 必背概念
1. `effective_timeout_ms = min(上游预算, 下游配置, 剩余deadline)` 的核心思想。

### 9.2 高频问答
1. 问：为什么最大重试次数还不够，还需要预算机制。  
答：仅限制单请求次数不能抑制系统级并发重试，预算可做全局速率约束。
2. 问：为什么要回传 `effective_timeout_ms`。  
答：便于确认预算是否真实生效，支持排查“看似超时配置正确却仍超时”的问题。
3. 问：预算耗尽为什么优先返回 429。  
答：429 能明确表达“请求被治理策略限流”，区别于上游服务实际故障。
