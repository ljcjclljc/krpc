## 4. 核心能力说明

### 4.1 RPC 抽象层
W1 在 RPC 层建立了三类关键接口：
1. `IServiceDiscovery`：服务发现抽象。
2. `ILoadBalancer`：负载均衡抽象。
3. `IRpcClient`：统一调用入口抽象。

配套能力：
1. 通过 `ClientInitOptions` 支持依赖注入。
2. 提供 `init_client/default_client` 全局装配与默认实例管理。
3. 提供内存版 `discovery + round-robin lb`，保证本地可运行。

### 4.2 配置快照与版本治理
W1 在 infra 层建立了最小闭环配置治理：
1. 快照读取：`snapshot()`。
2. 新版本更新：`update_if_newer()`。
3. 历史回滚：`rollback_to(version)`。
4. 配置中心刷新入口：`refresh_config_from_center()`。

### 4.3 工程化收益
1. 网关与 RPC 底层实现解耦，后续替换实现成本低。
2. 配置变更具备回滚路径，降低发布风险。
3. 单测可覆盖关键边界，降低回归成本。

1. 分层抽象先行，后续功能开发不需要反复重构调用边界。
2. 通过最小可运行桩实现，降低前期“接口有了但不可用”的风险。
3. 配置治理从 Day-1 就具备版本意识，为线上稳定性打底。
4. 把“可替换性”写成接口与测试，而不是口头约定。

### 8.2 高频问答
1. 问：为什么 W1 要先做抽象而不是直接写业务逻辑。  
答：先确定跨层契约可以降低后续迭代的重构成本，并让测试先行。
2. 问：`IRpcClient` 的价值是什么。  
答：网关只依赖统一调用接口，不感知 discovery/lb/传输实现细节。
3. 问：配置回滚为什么必须在早期提供。  
答：配置是高频变更面，回滚能力是稳定性的底线保障。

3. 能否给出一条“抽象错误导致耦合”的反例并说明 W1 如何规避。
