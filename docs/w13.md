# W13 帮助文档（自适应负载均衡 + 熔断降级 + 灰度分流）

## 1. 目标与范围
W13 目标是在 W10 基线调用链上补齐三类稳定性与发布治理能力：
1. 基于实时状态的自适应负载均衡（CPU/内存/QPS/延迟加权）
2. 熔断状态机（closed/open/half-open）与降级策略（缓存兜底/静态兜底）
3. 灰度发布与流量染色（按标签/用户/比例分流）

## 3. 核心设计说明

### 3.1 自适应负载均衡（AdaptiveLoadBalancer）
默认负载均衡从轮询升级为 `AdaptiveLoadBalancer`，按以下信号计算风险并选最大得分节点：
1. CPU 利用率
2. 内存利用率
3. QPS
4. 延迟
5. 失败 EWMA 与并发饱和度（附加惩罚项）

计算方式：
1. `risk = w_cpu*cpu + w_mem*mem + w_qps*qps_penalty + w_latency*latency_penalty + failure_penalty + saturation_penalty`
2. `score = 1 - risk`
3. 选择 `score` 最高节点

实时状态来源：
1. `ServiceNode` 外部上报指标（`cpu_utilization/memory_utilization/qps/latency_ms`）
2. 调用反馈更新的 EWMA（延迟、失败率、瞬时 QPS）

### 3.2 熔断状态机与恢复
每个节点维护独立熔断状态：
1. `closed`：正常放流
2. `open`：在 `open_window` 内拒绝选中
3. `half-open`：窗口到期后允许少量探测请求

状态流转：
1. `closed -> open`：连续失败达到阈值
2. `open -> half-open`：打开窗口到期
3. `half-open -> closed`：探测成功达到阈值
4. `half-open -> open`：探测失败立即回开路

### 3.3 降级策略
当上游失败且满足降级条件时，按顺序执行：
1. 缓存兜底：命中 `service|method`（或 `x-fallback-cache-key`）缓存直接返回
2. 静态兜底：返回 `x-fallback-static` 配置内容

响应侧可观测字段：
1. `degraded`
2. `degrade_strategy`（`cache`/`static`）
3. `selected_endpoint`
4. `traffic_lane`
5. `circuit_state`

### 3.4 灰度发布与流量染色
支持三种分流路径：
1. 标签直达：`x-traffic-tag=stable|gray`
2. 用户哈希比例：`x-user-id + x-gray-percent`
3. 标签自定义：`x-gray-tag/x-stable-tag`

网关会将相关 Header 透传到 `RpcRequest.metadata`，下游负载均衡直接消费。

## 6. 验收映射

### 6.1 验收项 1
节点流量分配可随实时状态动态收敛，故障节点可被自动熔断。

对应测试点（`w13_adaptive_lb_circuit_breaker_gray_test`）：
1. 高负载节点与低负载节点并存时，请求集中到低负载节点（收敛）
2. 设置 `x-gray-percent/x-user-id` 或 `x-traffic-tag` 后命中对应 lane
3. 故障节点连续失败后进入开路，流量自动切换到健康节点

### 6.2 验收项 2
过载/故障场景下可自动降级并快速恢复，避免级联雪崩。

对应测试点：
1. 失败后命中缓存兜底（`degrade_strategy=cache`）
2. 无缓存时命中静态兜底（`degrade_strategy=static`）
3. `open -> half-open -> closed` 恢复链路验证通过
