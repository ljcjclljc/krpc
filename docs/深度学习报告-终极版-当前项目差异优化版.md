# 第一部分：项目概述

## 1.1 项目背景与定位

### 1.1.1 项目定位（当前仓库）
当前项目是一个“网关 + RPC 调用治理 + 一致性元数据 + 运行时调度”一体化工程，不再只是单一 WebServer 练习项目。

核心目标：
1. 在入口层承接高并发 HTTP 请求并稳定退化。  
2. 在 RPC 层提供超时、重试预算、熔断、灰度、降级等治理能力。  
3. 在基础设施层提供配置热更新/回滚、TLS/mTLS 轮转、Raft 一致性元数据能力。  
4. 在工程层形成可测试、可观测、可发布基线。

### 1.1.2 相对原终极版的关键差异
原终极版以 `ucontext + ZooKeeper` 为主线；当前项目已演进为：
1. 协程底层：`boost::context::fiber`（不是 ucontext）。  
2. 元数据一致性：`Raft` 内建替代外部 ZooKeeper 强依赖。  
3. 网关能力：新增 Router/Servlet、鉴权拦截、错误码映射、Trace 与路由分位数。  
4. RPC 治理：新增自适应负载均衡、熔断、灰度分流、降级策略、重试预算。  
5. 稳定性：新增配置热更新与分钟级回滚、TLS/mTLS 轮转兜底。  
6. 工程化：W1-W18 文档、测试、压测、发布冻结流程完整。

## 1.2 整体架构设计

### 1.2.1 当前分层架构

```text
┌─────────────────────────────────────────────────────────────┐
│                         应用入口                             │
│                 apps/gateway/main.cpp                       │
├─────────────────────────────────────────────────────────────┤
│                         网关层                               │
│  ReactorHttpServer + HttpGateway + Router/Auth/Trace        │
├─────────────────────────────────────────────────────────────┤
│                         RPC 层                               │
│  IRpcClient + KrpcChannel + AdaptiveLB/CB/Gray/RetryBudget │
├─────────────────────────────────────────────────────────────┤
│                       Runtime 层                             │
│  Coroutine + Scheduler(M:N) + IOManager + Timer + Hook      │
├─────────────────────────────────────────────────────────────┤
│                        Infra 层                              │
│  Config Snapshot/Hot Reload + MemoryPool + StructuredLog    │
│  + Raft(Consensus/Storage/RPC)                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.2.2 启动顺序
`main()` 按依赖顺序初始化：
1. `infra::init_infra()`  
2. `runtime::init_runtime()`  
3. `net::init_network()`  
4. `gateway::init_gateway()`  
5. `client::init_client()`

该顺序保证：配置和运行时优先就绪，再启动网络入口和业务调用链。

# 第二部分：基础知识深度讲解

## 2.1 并发模型与工程化落地

### 2.1.1 当前并发模型
当前项目采用“协程 + M:N 调度 + epoll 事件驱动 + Hook 协程化阻塞调用”组合模型。

并发核心：
1. 协程作为轻量执行单元。  
2. 调度器将大量协程映射到有限 worker 线程。  
3. IO 事件就绪与定时器到期驱动协程恢复。  
4. 阻塞系统调用在 Hook 路径转为可让出执行权的异步等待。

### 2.1.2 相对原终极版差异
1. 原文重点是 `ucontext`，当前实现是 `boost::context`。  
2. 原文 tickle 以 `pipe` 叙述为主，当前工程采用 `eventfd` 唤醒。  
3. 当前工程更强调“治理+可观测+回滚”而不仅是“并发性能”。

## 2.2 协程底层机制（当前实现）

### 2.2.1 当前实现选型
- 头文件：`include/rpc/runtime/coroutine.h`  
- 实现：`src/runtime/coroutine.cpp`

协程对象基于 `boost::context::fiber`，并维护原子状态机：
1. `READY`  
2. `RUNNING`  
3. `WAITING`  
4. `TERM`

### 2.2.2 为什么这与原文不同
原终极版大量讲 `getcontext/makecontext/swapcontext`。当前项目在工程代码里不使用这套 API，面试与复盘时应改为讲 `boost::context fiber` 的切换语义和状态机约束。

## 2.3 IO 多路复用与 Reactor

### 2.3.1 核心实现点
- `src/runtime/io_manager.cpp`：runtime IO 事件循环。  
- `src/net/reactor_http_server.cpp`：HTTP Reactor 主循环。

关键点：
1. 均基于 `epoll`。  
2. 唤醒机制使用 `eventfd`。  
3. IO 与 timer 统一等待路径。  
4. 通过调度器执行回调/恢复逻辑。

### 2.3.2 相对原终极版差异
1. 原文强调“IOManager + Fiber 直接恢复”的叙事，当前项目在 runtime 与 net 层已分化为更工程化的回调调度模型。  
2. 当前新增大量并发控制与指标字段（高水位、429/503 分类、syscall 计数）。

## 2.4 RPC 基础与治理

当前项目 RPC 不再停留在“序列化 + 发现 + 连接池”基础能力，已扩展到治理：
1. 超时分层预算。  
2. 重试预算（滑动窗口 token）。  
3. 自适应 LB。  
4. 熔断状态机。  
5. 灰度分流与降级。

### 2.4.1 协议层
- `proto/krpc.proto`：请求/响应模型，`map<string,string> metadata`。  
- `src/rpc/krpc_channel.cpp`：Raw/Protobuf 编解码 + sync/async 调用。

### 2.4.2 一致性元数据
原文写 ZooKeeper。当前项目是：
- `proto/raft.proto`  
- `src/infra/raft/raft_cluster.cpp`  
- `src/infra/raft/raft_storage.cpp`  
- `src/infra/raft/raft_rpc.cpp`

# 第三部分：协程模块深度解析

## 3.1 模块概述

### 3.1.1 模块位置
- 接口：`include/rpc/runtime/coroutine.h`  
- 实现：`src/runtime/coroutine.cpp`

### 3.1.2 核心职责
1. 管理协程生命周期状态。  
2. 提供 `resume/yield/yield_waiting`。  
3. 提供当前协程获取与静态让出入口。

## 3.2 数据在协程中的处理流程

### 3.2.1 生命周期
1. 创建时 `READY`。  
2. 调度 `resume` 后进入 `RUNNING`。  
3. 主动 `yield` 回到 `READY`。  
4. 等待外部事件时 `yield_waiting` 进入 `WAITING`。  
5. 执行结束进入 `TERM`。

### 3.2.2 WAITING 语义
新增 `WAITING` 是当前项目的重要差异点。它让 IO 等待态与普通可运行态显式分离，避免误重入队。

## 3.3 核心数据结构

### 3.3.1 Coroutine 关键字段
1. `id_`：协程 ID。  
2. `callback_`：协程回调。  
3. `state_`：原子状态。  
4. `fiber_` / `caller_fiber_`：上下文句柄。  
5. `stack_size_`：栈配置。

### 3.3.2 与原终极版 Fiber 数据结构差异
1. 当前是 `Coroutine` 而非原文 `Fiber` 命名体系。  
2. 当前以 `boost::context` 对象为核心，不是 `ucontext_t`。  
3. 当前恢复逻辑与调度器深度耦合，强调跨线程安全状态转换。

## 3.4 协程创建与回收

1. 调度器负责分配 ID 并持有协程对象。  
2. 协程结束后由调度器回收并更新 `completed_count`。  
3. ID 可复用，降低长期运行对象编号膨胀。

## 3.5 协程切换关键路径

### 3.5.1 resume
`READY -> RUNNING`，切到协程上下文执行用户逻辑。

### 3.5.2 yield
`RUNNING -> READY`，让出执行权。

### 3.5.3 yield_waiting
`RUNNING -> WAITING`，等待 IO/Timer 事件恢复。

## 3.6 当前实现优缺点

### 3.6.1 优点
1. 状态语义清晰，非法切换更易检测。  
2. 与 IOManager/Timer/Hook 协同一致。  
3. 更贴近工程可维护性。

### 3.6.2 可继续优化点
1. 更细粒度调度可视化。  
2. 针对极端高并发场景的栈与上下文开销继续压缩。

# 第四部分：调度器模块深度解析

## 4.1 模块概述

### 4.1.1 模块位置
- 接口：`include/rpc/runtime/scheduler.h`  
- 实现：`src/runtime/scheduler.cpp`

### 4.1.2 核心职责
1. 协程调度与生命周期管理。  
2. M:N 线程池运行。  
3. WAITING 协程恢复。  
4. 运行态指标采集。

## 4.2 数据在调度器中的流转

### 4.2.1 任务提交
1. `schedule(callback)` 分配协程 ID。  
2. 协程进入队列。  
3. `pending_tasks` 增加。

### 4.2.2 执行与回收
1. worker 取任务执行。  
2. 根据状态决定重入队或回收。  
3. 回收后更新 `completed_count`。

## 4.3 M:N 调度模型（当前实现）

### 4.3.1 队列结构
1. 每 worker：`local_fast_queue` + `local_queue`。  
2. 全局：`global_queue`。  
3. 支持窃取：空闲 worker 从其他 worker 窃取。

### 4.3.2 并发语义
1. `resume(id)` 允许 WAITING 协程恢复。  
2. RUNNING 协程提前恢复请求通过 `pending_resumes_` 延后处理。

## 4.4 idle 行为与可观测
当前实现支持：
1. `idle_switch_count`。  
2. `steal_count`。  
3. enqueue/dequeue 分来源计数。  
4. 状态锁等待纳秒级统计。

## 4.5 相对原终极版差异总结
1. 原文调度器更教学化；当前是生产化剖面友好实现。  
2. 当前实现对竞态处理更完整。

# 第五部分：IO 调度器深度解析

## 5.1 模块概述

### 5.1.1 模块位置
- runtime IO：`include/rpc/runtime/io_manager.h` + `src/runtime/io_manager.cpp`  
- net Reactor：`include/rpc/net/reactor_http_server.h` + `src/net/reactor_http_server.cpp`

### 5.1.2 核心职责
1. 维护 fd 事件注册/触发。  
2. 在事件触发后调度回调执行。  
3. 与 timer 融合超时等待。  
4. 对外提供协程恢复入口。

## 5.2 数据在 IOManager 中的流转

### 5.2.1 事件注册
1. `add_event(fd, Read/Write, callback)` 绑定事件上下文。  
2. 维护 epoll interest 集。  
3. 事件线程阻塞于 `epoll_wait`。

### 5.2.2 事件处理
1. 就绪事件返回。  
2. `trigger_events` 取出并解绑上下文。  
3. 通过调度器执行回调。  
4. 需要时恢复 WAITING 协程。

## 5.3 Epoll 事件循环

1. 计算 `next_timeout_ms` 作为 `epoll_wait` 超时。  
2. 到期 timer 回调通过调度器执行。  
3. 停止、新增更早 timer、取消 timer 时通过 `eventfd` 唤醒。

## 5.4 唤醒机制（当前项目）

### 5.4.1 当前实现
当前项目使用 `eventfd` 唤醒，而不是原文主线中的 `pipe tickle`。

### 5.4.2 工程收益
1. 语义更直接。  
2. 与 epoll 集成更稳定。  
3. 代码路径统一于 runtime/net 两层。

## 5.5 对比分析

### 5.5.1 优点
1. IO 与 Timer 真正统一等待点。  
2. 恢复与回调模型可测试、可观测。  
3. 更贴近生产实现。

### 5.5.2 可改进点
1. 更细粒度 fd 级延迟分析。  
2. 进一步压缩高频 epoll_ctl 路径开销。

# 第六部分：Hook 机制深度解析

## 6.1 模块概述

### 6.1.1 模块位置
- 头文件：`include/rpc/runtime/hook.h`  
- 实现：`src/runtime/hook.cpp`

### 6.1.2 核心职责
1. 拦截阻塞调用并协程化等待。  
2. 保持非 Hook 路径原生语义。  
3. 通过 TLS 开关控制影响范围。

## 6.2 数据在 Hook 中的流转

### 6.2.1 Hook 生效条件
1. `hook_enabled == true`（线程本地）。  
2. 当前存在运行协程。  
3. 绑定了可用 IOManager。

### 6.2.2 调用链
1. 调用 `read/write/recv/send/connect/accept/sleep`。  
2. 若命中 `EAGAIN/EWOULDBLOCK/EINPROGRESS`，注册事件并 `yield`。  
3. 事件到达后恢复继续重试。

## 6.3 关键 Hook 函数

### 6.3.1 sleep Hook
- 转换为 timer + yield，不阻塞 worker 线程。

### 6.3.2 connect Hook
- 处理 EINPROGRESS，等待写事件，就绪后 `getsockopt(SO_ERROR)` 判定连接结果。

### 6.3.3 accept Hook
- 读事件驱动等待，保持统一 IO 等待模型。

## 6.4 与原终极版差异
1. 原文强调“全局 Hook”局限，当前实现是 TLS 线程级开关。  
2. 当前支持 connect 超时处理与 accept hook。  
3. 当前 Hook 由 runtime 外观层可控启停。

# 第七部分：HTTP 服务器深度解析

## 7.1 模块概述

### 7.1.2 核心职责

2. header/body 超限快速失败（431/413）。

4. TraceId 透传与快照。  
5. 路由维度 metrics（error_rate / p50 / p90 / p99）。

## 7.4 相对原终极版差异
1. 原文“HttpServer”偏通用，当前项目是“ReactorHttpServer + HttpGateway”解耦结构。  
2. 当前项目入口治理能力显著增强（有界队列、限流、在途上限、缓冲上限）。

# 第八部分：完整数据流深度分析

## 8.1 HTTP 请求完整生命周期

### 8.1.1 宏观视图

```text
Client
  -> accept
  -> read/parse
  -> ingress queue
  -> worker process_request
  -> gateway route/auth
  -> rpc invoke
  -> map http status
  -> append response
  -> write
  -> keep-alive or close
```

### 8.1.2 read 微观数据流
1. 非阻塞读取进入连接缓冲。  
2. parser 按增量推进状态。  
3. 解析失败或超限直接错误响应。  
4. 成功后转换为 ingress task。

### 8.1.3 send 微观数据流
1. worker 生成完整 HTTP response 字符串。  
2. reactor write 逐步发送并推进 `write_offset`。  
3. 写空后根据 keep-alive 与连接状态决定保活或关闭。

## 8.2 RPC 调用完整生命周期

### 8.2.1 宏观视图

```text
HttpGateway::handle_rpc_invoke
  -> build RpcRequest(metadata/trace/timeout/gray)
  -> IRpcClient::invoke
  -> discovery list_nodes
  -> adaptive lb select
  -> circuit/retry/degrade decision
  -> KrpcChannel encode(raw/protobuf)
  -> net::invoke_tcp (tcp/tls)
  -> decode response
  -> feedback lb runtime state
  -> map http status and return
```

### 8.2.2 服务发现流程
1. 默认可走内存发现。  
2. W12 后可通过 `RaftServiceDiscovery` 适配替换。

### 8.2.3 协议与序列化
1. Raw 或 Protobuf 模式。  
2. Protobuf 失败返回明确错误码（encode/decode failed）。

### 8.2.4 网络发送与接收
1. 解析 endpoint 与有效超时。  
2. TCP 路径或 TLS 路径执行 send/recv。  
3. TLS/mTLS 支持 context 缓存与轮转重建。

### 8.2.5 治理闭环
1. RetryBudget 控制重试规模。  
2. 熔断状态根据反馈动态变化。  
3. 降级策略按配置/metadata 生效。  
4. 全过程可输出 trace、metrics 与结构化日志。

## 8.3 配置热更新与回滚数据流

```text
publish_config_snapshot / publish_config_patch
  -> ConfigRepository version++
  -> 运行中模块按 version 感知并刷新
  -> 异常时 rollback_config_to(anchor_version)
  -> 指标恢复验证
```

## 8.4 与原终极版差异
1. 原文全链路不包含 Raft/TLS 轮转/热更新回滚主路径。  
2. 当前全链路强调治理与运维可恢复性。

# 第九部分：面试问答模板

## 9.1 协程相关面试题

### Q1: 你们协程底层是 ucontext 吗？
答：不是，当前项目用 `boost::context::fiber`。协程状态是 `READY/RUNNING/WAITING/TERM`，其中 WAITING 专门表示外部事件等待态。

### Q2: WAITING 状态的价值是什么？
答：把“可运行”和“等待外部事件”显式分离，避免协程在等待 IO 时被误调度。

### Q3: 协程恢复竞态如何处理？
答：调度器对 RUNNING 协程的提前恢复请求会缓存，等进入 WAITING 后补发，保证状态转换合法。

## 9.2 调度器相关面试题

### Q1: M:N 是怎么落地的？
答：多 worker + 本地队列 + 全局队列 + 快队列 + 工作窃取；调度器维护 pending/active/alive/completed 等指标。

### Q2: 如何观测调度瓶颈？
答：看 `steal_count`、队列出入分布、`state_lock_wait_ns` 等剖面字段。

### Q3: idle 的作用是什么？
答：在无任务时维持 worker 生命周期并等待唤醒，保证线程池可持续处理新增任务。

## 9.3 Epoll 相关面试题

### Q1: epoll 在项目中怎么用？
答：runtime IOManager 与 net Reactor 都基于 epoll，并通过 eventfd 实现安全唤醒。

### Q2: 为什么不是 pipe tickle？
答：eventfd 语义更直接，减少额外 fd 管理复杂度，runtime 与 net 两层都统一使用。

### Q3: epoll timeout 怎么算？
答：由 TimerManager 计算最近超时，直接作为 epoll_wait timeout，形成 IO+Timer 单等待点。

## 9.4 Hook 机制面试题

### Q1: Hook 为什么不会污染所有线程行为？
答：Hook 开关是 TLS 线程本地变量，可作用域启停。

### Q2: connect Hook 为什么要处理 EINPROGRESS？
答：非阻塞 connect 的正常中间状态，需要等待写就绪后读取 SO_ERROR 判断最终结果。

### Q3: Hook 后 read 为什么“看起来异步”？
答：EAGAIN 时不是阻塞线程，而是注册事件并让出协程，事件到达后恢复重试。

## 9.5 RPC 相关面试题

### Q1: 你们怎么控制重试风暴？
答：`max_retries` + `RetryBudget` 双控制。预算耗尽直接返回 429，阻断级联放大。

### Q2: 负载均衡是纯轮询吗？
答：不是，默认是自适应 LB，综合 CPU/内存/QPS/延迟和失败反馈打分，还结合熔断状态机。

### Q3: 为什么要做灰度分流？
答：降低发布风险。当前支持标签直达和用户哈希比例分流，并可回滚。

## 9.6 高可用与稳定性面试题

### Q1: 如何保证热更新不重启？
答：配置中心快照版本化，运行时组件按版本刷新；异常可回滚到锚点版本。

### Q2: 证书轮转失败怎么办？
答：TLS context 重建失败时保留旧 context 服务，避免新证书错误导致全量失败。

### Q3: 为什么从 ZooKeeper 转向 Raft？
答：把一致性元数据能力内建，降低外部依赖耦合，统一配置与服务发现治理链路。

## 9.7 回答技巧（当前项目）

1. 不再以 `ucontext` 为主线，改讲 `boost::context + 状态机 + 调度语义`。  
2. 强调“治理闭环”：超时、重试预算、熔断、灰度、降级。  
3. 强调“运维闭环”：热更新、回滚、TLS 轮转、故障注入。  
4. 用具体字段说话：`effective_timeout_ms`、`attempts`、`blocked_by_circuit_breaker`、`rejected_rate_limited_429`。
