## 4. 关键设计

### 4.1 调用链打通（HTTP -> RPC -> HTTP）
1. `ReactorHttpServer` 继续负责 accept/read/write 与请求分发。
2. `/rpc/invoke` 路由在 `HttpGateway` 内部组装 `RpcRequest` 并调用 `IRpcClient::invoke`。
3. RPC 返回码通过 `map_rpc_code_to_http_status` 映射回 HTTP 语义并回写给客户端。

### 4.2 TraceId 生成与透传
1. 网关入口优先读取 `x-trace-id`；若缺失则生成 UUID（Boost UUID）。
2. 网关将 TraceId 写入 `RpcRequest.metadata`：
- `x-trace-id`
- `trace_id`
3. RPC 默认实现读取上述字段并输出 RPC 日志，日志中包含 `trace_id/service/method/endpoint/code/attempts`。
4. 网关日志同样输出 `trace_id/route/status/latency_ms`，可直接按 TraceId 关联网关与 RPC 日志。

### 4.3 路由维度指标上报
`HttpGateway` 新增路由指标聚合：
1. 维度：`METHOD + normalized_uri`（例如 `POST /rpc/invoke`）。
2. 计数：`requests`、`errors`（HTTP `>=400` 记为错误）。
3. 错误率：`error_rate = errors / requests`。
4. 延迟分位数：`p50/p90/p99`（毫秒）。

对外暴露：
1. `HttpGateway::route_metrics_snapshot()`
2. `ReactorHttpServer::route_metrics_snapshot()`

### 4.4 可观测快照
为便于验收和联调，新增：
1. `HttpGateway::last_trace_snapshot()`：最近一次网关请求的 Trace 快照。
2. `rpc::client::last_invoke_audit_snapshot()`：最近一次 RPC 调用审计快照。

## 5. Boost 使用说明（简化实现）
W10 在既有 Boost 体系上继续复用并新增：
1. `boost::uuid::random_generator` + `boost::uuids::to_string`：用于 TraceId 生成。
2. 已有 `Boost.Beast/Asio/JSON/Algorithm` 继续用于 HTTP 解析、收发与字符串处理，避免手写协议与解析细节。

## 6. 新增测试与验收映射
新增测试：`tests/unit/w10_gateway_rpc_trace_metrics_test.cpp`

覆盖点：
1. `HTTP -> RPC -> HTTP`：`/rpc/invoke` 返回 200 且回包包含上游 payload。
2. 显式 TraceId 透传：`x-trace-id` 从网关传至 RPC，网关快照与 RPC 快照一致。
3. 缺失 TraceId 自动生成：网关生成后透传至 RPC，且不为空。
4. 路由错误率：`POST /rpc/invoke` 在失败请求后 `errors >= 1` 且 `error_rate > 0`。
5. 路由延迟分位数：`POST /slow` 的 `p50/p90/p99` 可观测且大于阈值。

对应验收结论：
1. HTTP -> RPC -> HTTP 端到端可用。
2. 可按 TraceId 关联网关日志与 RPC 日志。
3. 路由维度延迟分位数与错误率可上报。
