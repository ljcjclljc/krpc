## 4. 具体改动与作用

### 4.1 IOManager / Epoll 事件循环
具体改动：
1. 增加 `IOManager`：封装 `epoll_create1/epoll_ctl/epoll_wait`。
2. 提供 `start/stop`：独立事件循环线程负责等待与分发事件。
3. 提供 `add_event/del_event/add_event_for_current`：支持回调或协程绑定。
4. 在事件触发后同步更新 epoll 监听掩码，避免解绑后仍被反复唤醒。

作用：
1. 将 IO 事件转换为调度器任务或协程恢复，形成统一入口。
2. 为后续 Hook 与超时机制提供稳定的 IO 基础能力。

### 4.2 FdContext 事件模型
具体改动：
1. 每个 FD 对应一个 `FdContext`，包含读/写两个 `EventContext`。
2. `EventContext` 记录回调或协程 ID，`active` 标记事件是否占用。
3. `trigger_events` 在锁内完成解绑与 epoll 更新，锁外完成回调/恢复。

作用：
1. 保证同一 FD 的同一事件一对一绑定，避免重复注册。
2. 在事件触发时完成“校验 + 清理 + 更新”，防止竞态回调。

### 4.3 协程等待态支持
具体改动：
1. 协程状态新增 `WAITING`，用于 IO 等待。
2. 新增 `yield_current_waiting()` 与 `try_mark_ready_from_waiting()`。

作用：
1. 等待 IO 的协程不会被误重入队列。
2. IO 触发时可通过 CAS 保证只恢复一次。

### 4.4 调度器恢复接口
具体改动：
1. 新增 `CoroutineScheduler::resume()`：仅允许 `WAITING -> READY`。
2. `execute_coroutine` 支持 `WAITING` 分支，不再重入队列。

作用：
1. IO 事件触发后可安全恢复协程。
2. 避免重复恢复或非法状态导致的竞态。

## 5. 竞态控制策略
1. 同一 FD 的注册、删除、触发统一使用 `FdContext` 互斥锁。
2. 触发路径在锁内完成解绑与 epoll 更新，锁外执行回调/恢复。
3. 协程恢复通过 `try_mark_ready_from_waiting()` 校验，避免重复恢复。

## 6. 新增测试与可行性验证
新增测试：`tests/unit/io_epoll_integration_test.cpp`

覆盖点：
